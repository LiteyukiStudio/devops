# AI 能力提案

## 1. 提案背景

平台当前主目标是把代码仓库、构建、镜像站、运行环境、网关、域名和发布记录打通。AI 能力不应该直接塞进第一阶段核心链路，否则会让 MVP 复杂度失控。

建议将 AI 作为独立能力层设计：

- 主平台继续负责项目、应用、构建、部署、权限和审计。
- AI 能力单独作为 `ai-service` 或 `agent-service` 演进。
- AI 服务通过平台 API、内部事件和受控工具调用平台能力。
- 平台可以向 AI 服务暴露一套 API Skill / Tool Schema。
- AI 服务的执行结果必须回写平台，形成可审计、可回滚、可追踪的操作记录。

第一阶段只做提案和接口预留，不阻塞 CRUD、构建和部署主链路。

## 2. AI 能力目标

AI 能力的定位不是替代 DevOps 平台，而是降低用户在配置、排障、规范化和重复操作上的成本。

目标：

- 帮用户理解项目状态。
- 帮用户生成配置草稿。
- 帮用户解释构建和部署失败原因。
- 帮用户生成平台模块配置、Dockerfile、Ingress、环境变量模板。
- 帮用户根据仓库内容推荐部署方案。
- 帮用户自动执行低风险操作。
- 对高风险操作生成计划，并等待用户确认。
- 给平台运营者提供容量、安全、成本和异常分析。

非目标：

- 不让 AI 绕过权限系统。
- 不让 AI 直接操作 Kubernetes、Git、Registry。
- 不让 AI 自己保存明文密钥。
- 不让 AI 在没有审计记录的情况下执行变更。
- 不让 AI 在 MVP 主链路中成为必需依赖。

## 3. 可介入的产品场景

### 3.1 项目创建助手

用户输入项目用途，AI 生成：

- 项目名称和 slug。
- 默认环境：dev、staging、prod。
- 命名空间建议。
- 成员角色建议。
- 默认资源配额建议。
- 推荐 root domain 策略。

落地方式：

- AI 只生成草稿。
- 用户确认后由平台 API 创建项目。
- 所有创建动作写 AuditLog。

### 3.2 应用接入助手

用户授权仓库后，AI 分析仓库结构：

- 识别语言和框架。
- 判断是否已有 Dockerfile。
- 识别服务端口。
- 识别构建上下文。
- 识别包管理器。
- 识别健康检查路径。
- 识别是否需要数据库、Redis、对象存储等外部依赖。

适用文件：

- `Dockerfile`
- `docker-compose.yml`
- `package.json`
- `go.mod`
- `pyproject.toml`
- `requirements.txt`
- `pom.xml`
- `build.gradle`
- 平台 BuildSpec / 模块配置草稿。
- `helm/`
- `k8s/`

### 3.3 Dockerfile 生成与优化

AI 可以帮助：

- 生成 Dockerfile。
- 优化多阶段构建。
- 添加非 root 用户。
- 添加 `.dockerignore`。
- 解释镜像体积过大的原因。
- 推荐基础镜像。
- 检查是否误复制密钥、缓存、node_modules、dist 等内容。

安全要求：

- 只生成建议或 PR。
- 不直接覆盖用户文件。
- 对基础镜像、包安装命令、下载脚本进行风险提示。

### 3.4 模块配置助手

AI 根据仓库和目标镜像站生成：

- Build context。
- Dockerfile path。
- 镜像 tag 策略。
- 构建参数。
- 构建资源建议。
- 内网镜像站访问策略。
- BuildNetworkPolicy 草稿。

示例：

```yaml
build:
  dockerfile: Dockerfile
  context: .
  resources:
    cpuLimit: "1"
    memoryLimit: 2Gi
  network:
    mode: restricted
    allowPrivateCIDRHttps: true
```

### 3.5 工作流 YAML 可视化助手

平台已经规划 Builder 构建模板和可视化参数编辑。AI 可以介入：

- 从自然语言生成 BuildSpec。
- 从表单配置生成 BuildRun 参数。
- 解释现有模块配置。
- 检查危险步骤。
- 检查 secret 使用是否合理。

注意：

- 构建执行只走平台 Builder，不接入 GitHub Actions、Gitea Actions 或 Kubernetes Job Provider。
- 平台主路径是 Builder Agent + executor 容器 + BuildKit rootless。

### 3.6 构建失败诊断

AI 读取 BuildRun 信息：

- commit。
- Dockerfile。
- 构建参数。
- 最后 N 行日志。
- 镜像站错误。
- 网络策略拒绝事件。
- 资源限制和超时信息。

输出：

- 失败原因分类。
- 最可能原因。
- 修复建议。
- 可执行修复草稿。
- 需要人工确认的危险操作。

失败分类：

- 依赖安装失败。
- Dockerfile 路径错误。
- 构建上下文错误。
- registry 登录失败。
- registry push 失败。
- 网络策略阻断。
- DNS 解析失败。
- 资源不足。
- 超时。
- 基础镜像拉取失败。

### 3.7 部署失败诊断

AI 读取：

- Release。
- Deployment 状态。
- ReplicaSet。
- Pod events。
- container logs。
- probes。
- Service。
- Ingress/GatewayRoute。
- 证书状态。

输出：

- 发布失败原因。
- 是否建议回滚。
- 建议修改的端口、环境变量、资源限制、探针。
- 是否可能是镜像不可拉取、配置错误、CrashLoopBackOff、端口不匹配。

### 3.8 网关与域名助手

AI 可以帮助：

- 生成默认域名解释。
- 检查自定义域名 CNAME 是否配置正确。
- 解释证书申请失败原因。
- 判断是否可以使用 HTTP Challenge。
- 生成 DNS 配置提示。
- 解释 Traefik/Ingress 路由冲突。

### 3.9 密钥与配置助手

AI 可以帮助：

- 根据应用类型推荐配置项。
- 生成 ConfigMap/Secret 草稿。
- 检查配置项命名一致性。
- 检查是否把密钥误放到普通配置。
- 检查环境变量是否缺失。

限制：

- AI 不读取 Secret 明文。
- AI 只看到 secret key 名称、引用关系和校验状态。
- AI 不输出密钥值。

### 3.10 成本与资源助手

AI 基于 usage 数据：

- 分析 Project 资源消耗。
- 解释 credit 消耗趋势。
- 推荐 CPU/Memory request/limit。
- 找出长期空闲应用。
- 找出频繁失败的构建。
- 预测月度资源消耗。

### 3.11 安全助手

AI 可以辅助：

- 检查 Dockerfile 高风险命令。
- 检查构建网络策略是否过宽。
- 检查 webhook 配置是否缺少签名。
- 检查 Access Token 是否过期过长。
- 检查镜像来源是否可信。
- 检查是否使用 latest tag。
- 检查公开域名是否暴露管理面板。
- 检查高权限用户和项目成员变更。

### 3.12 平台运维助手

面向平台管理员：

- 集群健康摘要。
- Worker 队列积压解释。
- 构建失败率趋势。
- 镜像站错误率。
- 证书失败列表。
- 多集群容量建议。
- 异常审计事件摘要。

### 3.13 文档和模板助手

AI 可以生成：

- 项目 README 部署说明。
- Dockerfile。
- `.dockerignore`。
- 平台 BuildSpec / 模块配置草稿。
- Helm values 草稿。
- 环境变量文档。
- 运维 Runbook。
- 故障复盘摘要。

## 4. 推荐架构

### 4.1 独立 AI 服务

建议新增独立服务：

```text
web
  -> api
  -> ai-service
  -> model provider

api
  -> postgres
  -> redis/asynq
  -> worker

ai-service
  -> postgres read model
  -> vector store
  -> object storage
  -> platform tool gateway
  -> model provider
```

职责划分：

- `api`: 平台主业务，权限、审计、CRUD、构建、部署入口。
- `worker`: 执行构建、部署、同步、证书等异步任务。
- `ai-service`: 对话、Agent 编排、RAG、工具调用、计划生成。
- `tool-gateway`: 将平台能力暴露成受控工具，做权限、审计和确认。

### 4.2 为什么 AI 服务要独立

原因：

- AI 依赖变化快，不应污染核心 API。
- Agent 任务可能长时间运行，需要单独扩缩容。
- 模型调用需要独立限流、成本统计和审计。
- RAG、向量库、上下文缓存和工具调用链路与 CRUD API 生命周期不同。
- 后续可以替换 Google ADK、OpenAI Agents SDK、LangGraph 等框架。

### 4.3 服务间交互

推荐三种交互方式：

```text
同步查询:
web -> api -> ai-service -> api read/tool API

异步任务:
web -> api -> ai-service -> redis/asynq -> worker

事件驱动:
worker/api -> domain event -> ai-service -> insight/diagnosis
```

AI 不直接访问底层 Kubernetes、Git、Registry。所有动作必须走平台 API 或 Tool Gateway。

## 5. API Skill / Tool 设计

平台可以将能力抽象成 Skill：

```text
Skill
  - name
  - description
  - inputSchema
  - outputSchema
  - requiredPermissions
  - riskLevel
  - confirmationRequired
  - auditCategory
```

示例 Skill：

```text
project.list
project.createDraft
application.inspectRepository
application.generateConfigDraft
build.explainFailure
build.trigger
deployment.explainFailure
deployment.rollbackDraft
gateway.checkDomain
registry.testCredential
security.reviewBuildPolicy
cost.explainUsage
```

风险等级：

```text
read_only
draft_only
low_risk_write
high_risk_write
destructive
```

执行规则：

- `read_only`: 可直接执行，但必须受权限控制。
- `draft_only`: 只生成草稿，不写业务状态。
- `low_risk_write`: 可由用户授权后执行。
- `high_risk_write`: 必须二次确认。
- `destructive`: MVP 禁止 AI 直接执行，只允许生成操作计划。

## 6. MCP 兼容方案

可以为平台实现 MCP Server，让外部 AI Client 或内部 AI 服务调用平台能力。

MCP 暴露对象：

- Tools：执行平台动作。
- Resources：读取项目、应用、构建、发布、日志摘要。
- Prompts：提供固定诊断模板、配置生成模板、复盘模板。

示例 MCP Tools：

```text
liteyuki.projects.list
liteyuki.applications.inspect
liteyuki.builds.explain_failure
liteyuki.deployments.explain_failure
liteyuki.gateway.check_domain
liteyuki.security.review_project
```

MCP 安全要求：

- 每个 Tool 必须声明权限和风险等级。
- Tool 调用必须带用户身份或服务身份。
- 所有 Tool 调用写 AuditLog。
- 高风险 Tool 返回 confirmation request，不直接执行。
- 防止 prompt injection 诱导跨项目访问。
- 防止把 Secret 明文作为 Resource 暴露。

## 7. Google ADK 落地方案

Google Agent Development Kit 可以作为 AI 服务的一个候选 Agent 框架。根据官方文档，ADK 是开源、代码优先的 Agent 开发框架，支持工具、多 Agent、本地开发 UI，并可部署到 Cloud Run 或 Vertex AI Agent Engine。

### 7.1 推荐用途

ADK 适合承载：

- 多 Agent 协作。
- DevOps 诊断 Agent。
- 安全审查 Agent。
- 配置生成 Agent。
- 成本分析 Agent。
- 平台运维 Agent。

### 7.2 Agent 划分

```text
RootAgent
  - RouterAgent
  - RepositoryInspectorAgent
  - BuildDoctorAgent
  - DeploymentDoctorAgent
  - GatewayDoctorAgent
  - SecurityReviewerAgent
  - CostAdvisorAgent
  - TemplateWriterAgent
```

职责：

- `RouterAgent`: 判断用户意图并分派子 Agent。
- `RepositoryInspectorAgent`: 分析仓库结构和应用类型。
- `BuildDoctorAgent`: 解释构建失败。
- `DeploymentDoctorAgent`: 解释部署失败。
- `GatewayDoctorAgent`: 解释域名、证书、路由问题。
- `SecurityReviewerAgent`: 检查配置和策略风险。
- `CostAdvisorAgent`: 分析资源与 credit 消耗。
- `TemplateWriterAgent`: 生成配置、Dockerfile、BuildSpec 和文档草稿。

### 7.3 ADK 工具封装

ADK Agent 不直接操作平台数据库，而是调用 Tool Gateway：

```text
ADK Tool
  -> Tool Gateway HTTP API
  -> Platform API
  -> Permission Check
  -> AuditLog
  -> Domain Service
```

示例工具：

```python
def get_build_run(build_run_id: str) -> dict:
    ...

def get_build_log_tail(build_run_id: str, lines: int = 200) -> dict:
    ...

def create_application_config_draft(project_id: str, repository_id: str) -> dict:
    ...

def propose_rollback(release_id: str) -> dict:
    ...
```

### 7.4 ADK 部署方式

可选部署：

- 本地开发：使用 ADK dev UI 调试 Agent。
- 平台自托管：作为 `ai-service` 容器部署到本平台管理的 K8s/K3s。
- Google Cloud Run：轻量托管 Agent 服务。
- Vertex AI Agent Engine：托管运行、扩缩容和生产管理。

建议：

- MVP 后的第一版 AI 使用自托管 `ai-service`。
- 如果后续使用 Google Cloud 生态，可以增加 Agent Engine Provider。
- 不把 Google Cloud 绑定为唯一实现。

## 8. OpenAI Agents SDK 兼容方案

OpenAI Agents SDK 可以作为另一种 Agent Runtime。官方能力包括工具、handoff、guardrails、streaming 和 tracing，适合做多 Agent 编排和受控工具调用。

推荐用法：

- 使用 handoff 将任务转给 BuildDoctor、DeployDoctor、SecurityReviewer。
- 使用 guardrails 检查输入输出是否包含敏感信息或越权请求。
- 使用 tool approval 实现高风险操作确认。
- 使用 tracing 记录 Agent 决策链路。

适用场景：

- 需要 OpenAI 模型和工具生态时。
- 需要较强的 guardrails 和 tracing 时。
- 需要与 OpenAI Hosted MCP / File Search / Web Search 等能力整合时。

## 9. LangGraph 兼容方案

LangGraph 适合有明确状态机、人工确认和长任务恢复需求的流程。

适合场景：

- 部署失败诊断流程。
- 发布回滚决策流程。
- 安全审查流程。
- 需要 human-in-the-loop 的多步骤计划。
- 需要 durable execution 的长任务。

示例流程：

```text
collect_context
  -> diagnose
  -> propose_actions
  -> wait_for_human_approval
  -> execute_low_risk_actions
  -> summarize_result
```

## 10. RAG 与知识库

AI 服务需要一个知识层：

- 平台产品文档。
- 项目配置。
- 构建日志摘要。
- 发布记录。
- 故障复盘。
- 用户自定义 Runbook。
- Dockerfile 和 BuildSpec 模板。
- 常见错误知识库。

数据源：

- `docs/`
- 项目 README。
- BuildRun/Release 摘要。
- AuditLog。
- 用户上传的团队文档。

向量库候选：

- PostgreSQL + pgvector。
- Qdrant。
- Milvus。
- Weaviate。

建议第一阶段：

- 使用 PostgreSQL + pgvector。
- 只索引非敏感文本。
- Secret 明文不进入向量库。
- 日志只索引摘要和脱敏片段。

## 11. 安全与治理

### 11.1 权限

AI 的权限必须小于等于用户权限。

```text
effectivePermissions = userPermissions ∩ aiServicePolicy ∩ toolPolicy
```

### 11.2 审计

所有 AI 行为需要记录：

- conversation id。
- agent id。
- model provider。
- model name。
- tool name。
- input hash。
- output hash。
- user id。
- project id。
- risk level。
- confirmation status。
- execution result。

### 11.3 Secret 保护

规则：

- AI 不读取 Secret 明文。
- AI 只读取 secret key 名称、挂载路径、引用对象和校验状态。
- 构建日志进入 AI 前必须脱敏。
- Access Token 只展示一次，不能进入 AI 上下文。

### 11.4 Prompt Injection 防护

风险来源：

- 仓库 README。
- 构建日志。
- 用户上传文档。
- issue/commit message。
- 外部网页。

防护：

- 外部内容作为不可信上下文。
- 工具调用由系统策略控制，不由上下文文本授权。
- Agent 输出计划和工具调用前做策略检查。
- 高风险操作必须用户确认。

### 11.5 成本控制

需要记录：

- token 用量。
- 模型费用。
- Agent 运行时间。
- 工具调用次数。
- 向量检索次数。
- 每个 Project 的 AI credit 消耗。

## 15. 分阶段路线

### Phase 0：提案与预留

- 完成 AI 能力提案。
- 在 API/权限/审计模型中预留 AI 行为记录。
- 不进入 MVP 主链路。

### Phase 1：只读诊断

- BuildRun 失败解释。
- Release 失败解释。
- 项目状态摘要。
- 不执行任何写操作。

### Phase 2：草稿生成

- 生成 Dockerfile 草稿。
- 生成 BuildSpec 草稿。
- 生成 BuildNetworkPolicy 草稿。

### Phase 3：受控工具调用

- 引入 Tool Gateway。
- 支持低风险写操作。
- 高风险操作必须确认。
- 完整 AuditLog。

### Phase 4：Agent Runtime

- 引入 Google ADK / OpenAI Agents SDK / LangGraph Provider。
- 支持多 Agent 协作。
- 支持长期任务、人工确认和 trace。

### Phase 5：MCP Server 与外部生态

- 暴露平台 MCP Server。
- 支持外部 AI Client 调用平台只读能力。
- 支持内部 AI 服务调用 MCP Tools。
- 支持 Skill 市场或模板市场。

## 16. MVP 后第一版建议

最小可落地 AI 版本：

- 单独 `ai-service`。
- 一个 `BuildDoctorAgent`。
- 一个 `DeploymentDoctorAgent`。
- 一个 `TemplateWriterAgent`。
- 只读平台上下文。
- 只生成草稿，不执行写操作。
- 使用 PostgreSQL 保存 conversation/run/tool call。
- 日志进入 AI 前脱敏。
- 前端在 BuildRun/Release 页面增加“AI 诊断”按钮。

这个版本价值高、风险低，不会破坏平台主链路。
