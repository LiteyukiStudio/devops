# 场景测试问题记录

本文档用于记录每次用户场景测试发现的问题、影响范围、修复动作和复测状态。每次执行新的场景测试后，都必须追加一条测试批次记录；如果测试中发现问题，也必须在对应批次下补充问题条目。

## 记录规则

- 每次场景测试都新增一个批次，标题格式为 `YYYY-MM-DD 场景名称`。
- 每个问题都记录：现象、影响、处理状态、修复动作、复测结果。
- 修复完成但未复测时，状态保持 `待复测`。
- 确认不是缺陷但会影响体验时，状态使用 `体验待优化`。
- 被产品决策接受的限制，状态使用 `已接受`，并记录原因。
- 涉及后续开发计划的问题，同步回填 `TODO.md`。

## 状态说明

| 状态 | 含义 |
| --- | --- |
| 待处理 | 已确认问题，尚未开始修复 |
| 修复中 | 已开始改动，但未完成验证 |
| 待复测 | 已修复，等待场景回归 |
| 已复测 | 已通过同类场景验证 |
| 体验待优化 | 不阻断主路径，但影响用户体验 |
| 已接受 | 当前阶段接受该限制，不安排立即修复 |

### 修复记录

- 项目空间工作台新增“仓库绑定”tab，可从项目空间内直接创建和维护 RepositoryBinding。
- 项目空间删除后会收敛分页并主动刷新项目空间分页查询，避免空状态和总数不一致。
- 集群资源页新增平台管理资源删除入口，并通过后端校验 `liteyuki.devops/managed-by` 标签后删除。
- Pod 摘要补充容器 waiting/terminated reason 和 Pod condition，部署卡住时不再只显示 `ready 0/1`。
- 应用基础表单恢复服务端口字段，用于正确生成 Deployment/Service/Ingress 端口。
