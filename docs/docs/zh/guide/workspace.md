# 连接集群和镜像站

平台跑起来以后，不需要马上配置所有外部系统。按这条顺序走，最不容易绕路。

```text
运行集群 -> 镜像站 -> 部署配置 -> 访问入口
```

## 1. 配置运行集群

运行集群是应用最终部署的位置。可以是 Kubernetes，也可以是轻量的 K3s。

第一次联调建议准备一个可测试的集群，并确认 kubeconfig 能从 API / worker 容器访问。

删除运行集群前，需要先把引用它的部署配置迁移到其他集群或删除；平台不会在删除集群时自动删除部署配置。

## 2. 配置镜像站

镜像站用于保存构建产物，也可以用于直接部署已有镜像。

如果只是体验部署流程，可以先使用已有镜像；如果要跑完整构建链路，再配置推送凭据。

## 3. 创建部署配置

部署配置回答“这个应用怎么发布”：

- 用已有镜像，还是从仓库构建。
- 使用哪个运行集群。
- 属于开发、测试、预发还是生产阶段。
- 运行副本数、CPU 和内存规格。
- 构建 Job 的 CPU 和内存规格。
- 服务端口是多少。
- 是否需要环境变量、Secret、配置文件和数据卷。

如果只是先验证平台，选择“已有镜像”最快；确认集群和访问入口没问题后，再接 Git Provider 和自动构建。

## 4. 创建访问入口

访问入口把域名、路径、TLS 和后端服务连起来。创建后，在页面里查看状态，再用浏览器或 `curl` 验证。

## 5. 再开启自动化

第一条链路跑通后，再逐步开启：

- Git Provider 和 Git 账号授权。
- 仓库绑定和 Webhook。
- 自动构建。
- 构建成功后自动部署。
- 自定义域名和 HTTPS。
