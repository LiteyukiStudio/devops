# 连接集群和镜像站

平台能打开以后，先别急着把所有外部系统都接上。按下面的顺序准备，遇到问题时更容易判断是哪一步出了错。

```text
运行集群 -> 镜像站 -> 部署配置 -> 访问入口
```

## 1. 配置运行集群

运行集群就是应用真正运行的地方，可以使用 Kubernetes，也可以使用更轻量的 K3s。

第一次联调建议准备一个可测试的集群，并确认 kubeconfig 能从 API / worker 容器访问。

删除运行集群前，需要先把引用它的部署配置迁移到其他集群或删除；平台不会在删除集群时自动删除部署配置。

## 2. 配置镜像站

镜像站用于保存构建产物，也可以用于直接部署已有镜像。

如果只是体验部署流程，可以先使用已有镜像；如果要跑完整构建链路，再配置推送凭据。

## 3. 创建部署配置

部署配置用来说明“这个应用要怎样构建和运行”：

- 用已有镜像，还是从仓库构建。
- 使用哪个运行集群。
- 属于开发、测试、预发还是生产阶段。
- 运行副本数、CPU 和内存规格。
- 构建 Job 的 CPU 和内存规格。
- 服务端口是多少。
- 是否需要环境变量、Secret、配置文件和数据卷。

如果只是先验证平台，选择“已有镜像”最快；确认集群和访问入口没问题后，再接 Git Provider 和自动构建。

## 4. 创建访问入口

访问入口把域名、路径和 TLS 指向已经部署好的服务。创建后先看页面状态，再用浏览器或 `curl` 发起一次真实请求。

## 5. 再开启自动化

第一次手动部署成功后，再逐步开启这些自动化：

- Git Provider 和 Git 账号授权。
- 仓库绑定和 Webhook。
- 自动构建。
- 构建成功后自动部署。
- 自定义域名和 HTTPS。
