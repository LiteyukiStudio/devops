## 一、API 启动与路由设计

### 1.1 启动阶段

1. `cmd/api/main.go` 读取配置。
2. 初始化数据库连接。
3. 执行 SQL migration，保证 schema 到达最新版本。
4. 创建 Gin router。
5. 注册中间件：日志、恢复、错误响应、安全响应头、CORS、CSRF Origin Guard。
6. 注册 `/api/v1` 业务路由和 `/healthz`。
7. 在 embed_web 构建模式下注册内嵌 SPA。

### 1.2 请求处理规则

- Handler 负责绑定参数、读取当前用户、调用权限校验和返回响应。
- 权限失败返回稳定业务错误。
- Cookie 会话请求需要校验 Origin；Bearer Token 请求走 token scope。
- API 返回稳定 code，前端负责 i18n 文案映射。

## 二、认证详细设计

### 2.1 本地登录

1. 用户提交账号和密码。
2. 后端按用户查找密码 hash。
3. 密码校验成功后创建 session。
4. session cookie 使用 HttpOnly，前端只通过 `/users/me` 获取当前用户。

### 2.2 OIDC 登录

1. 前端跳转到 `/auth/oidc/:providerId/start`。
2. 后端生成 state 并跳转 OIDC Provider。
3. 回调 `/auth/oidc/callback` 校验 state、token 和 userinfo。
4. 准入策略校验邮箱、组、邀请和域名。
5. 绑定现有用户或拒绝登录。

### 2.3 Access Token

- Token 明文仅创建时展示一次。
- 数据库保存 token hash、scope、过期时间和撤销时间。
- API 根据路径和方法匹配 scope，未知接口默认拒绝。

## 三、构建详细设计

### 3.1 BuildRun 创建

1. 用户选择部署配置触发构建。
2. API 从 DeploymentTarget 复制 Dockerfile、构建上下文、镜像仓库、资源请求等快照。
3. 创建 BuildRun，状态为 `queued`。
4. 创建 BuildJob，状态为 `queued`。
5. 投递 Asynq 构建任务。

### 3.2 Worker 构建执行

1. Worker 加载 BuildJob 和 BuildRun。
2. 校验项目、应用、部署配置、仓库绑定和镜像站凭据。
3. 渲染构建变量和密钥，密钥以 Kubernetes Secret 或执行环境注入。
4. 生成 Kubernetes Job 规格。
5. 配置 NetworkPolicy，按 `BUILD_EGRESS_MODE` 控制出站访问。
6. 启动 BuildKit rootless 执行构建和推送。
7. 持续采集日志写入 BuildLog。
8. 成功时记录 image ref、digest、资源用量和 ContainerImage。
9. 失败时记录 message 和日志。

### 3.3 自动部署

若 DeploymentTarget 开启 `autoDeploy`，BuildRun 成功后触发 `build_auto_deploy`：

1. 检查分支和 tag pattern。
2. 创建 Release，引用 BuildRun 和 image ref。
3. 投递部署任务。

## 四、部署详细设计

### 4.1 Release 创建

Release 表示一次部署或回滚动作，包含镜像、部署配置、revision、状态和日志。手动发布、自动发布和回滚都使用 Release 记录。

### 4.2 Kubernetes 资源生成

部署资源由 Kubernetes Provider 生成：

- Namespace：项目空间隔离。
- Deployment：容器镜像、副本、资源请求、环境变量和卷挂载。
- Service：使用部署配置 service port。
- ConfigMap/Secret：运行配置和敏感配置。
- PVC：启用数据保留时创建。
- Ingress：由 GatewayRoute 生成。

资源命名使用内部 ID 派生，避免用户输入的中文、空格或超长名称破坏 Kubernetes 约束。

### 4.3 Rollout 观察

Worker apply 资源后等待 Deployment rollout，在超时前轮询状态。成功则标记 Release `succeeded`，失败则写入 ReleaseLog 并标记 `failed`。

## 五、应用市场详细设计

### 5.1 模板结构

模板由 `internal/appstore/templates.json` 内嵌，包含：

- 基础信息：id、slug、name、description、category、icon。
- 来源信息：officialWebsite、officialRepository、image、version。
- 默认运行参数：servicePort、replicas、CPU、memory。
- 配置注入：env、secretEnv、configFiles、secretFiles。
- 用户参数：values。

### 5.2 模板安装流程

1. API 读取模板并渲染用户输入。
2. 创建 Application。
3. 创建 DeploymentTarget。
4. 写入密钥引用和配置文件。
5. 记录 AppTemplateInstallation。
6. 可选创建首个 Release。

## 六、计费详细设计

### 6.1 账本模型

- `UserWallet` 保存用户余额。
- `BillingRateRule` 保存计量项价格。
- `BillingUsageRecord` 保存用量，粒度包含 project、application、resource、meter。
- `BillingLedgerEntry` 保存余额变化，使用 idempotency key 防止重复入账。

### 6.2 归属规则

项目空间有 `billingOwnerUserId`。转移前产生的用量归旧 owner，转移后产生的用量归新 owner。充值和补偿直接进入用户账户，不再归属项目空间。

## 七、前端详细设计

### 7.1 页面结构

前端以 React Router 组织主要页面：

- Dashboard
- Projects / ProjectWorkspace
- Applications
- AppTemplates
- CodeRepositories
- Registries
- Clusters
- Billing
- Settings

### 7.2 状态与数据

- TanStack Query 负责服务端状态、缓存、刷新和失效。
- API client 统一处理错误、认证和 JSON。
- i18next 管理中英文文案。
- `ContentTabs`、`DataList`、`StatusBadge` 等公共组件统一页面体验。

### 7.3 交互约束

- 列表默认使用 DataList。
- 状态必须使用语义颜色 badge。
- 复杂表单字段提供说明提示。
- Secret 编辑时不展示原值，留空表示不修改。
