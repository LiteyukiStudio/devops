## 一、系统架构

### 1.1 架构风格

系统采用模块化单体加多进程部署：

- `cmd/api`：HTTP API、认证、CRUD、Webhook、任务投递、嵌入式前端静态文件。
- `cmd/worker`：异步任务执行，包括构建、部署、证书、资源清理、计费结算。
- PostgreSQL：业务数据、状态、账本、日志索引和配置。
- Redis + Asynq：异步任务队列。
- Kubernetes/K3s：构建执行环境和应用运行环境。
- Rspress 文档站：面向用户和开发者的在线文档。

### 1.2 架构图

```text
Browser
  -> API / embedded SPA
     -> PostgreSQL
     -> Redis / Asynq
     -> Git provider APIs
     -> Registry provider APIs
     -> Kubernetes provider APIs

Worker
  -> Redis / Asynq
  -> PostgreSQL
  -> Kubernetes Build Job / Deployment / Ingress
  -> Registry push / tag query
```

### 1.3 分层原则

| 层级 | 典型目录 | 职责 |
| --- | --- | --- |
| HTTP 层 | `internal/api` | 路由、认证入口、请求解析、响应映射 |
| 模型层 | `internal/model` | GORM 数据模型和序列化字段 |
| 数据访问 | `internal/repository` | 共享查询、归属绑定和复杂数据读取 |
| 业务服务 | `internal/service`、`internal/billing` | 权限策略、计费、令牌、业务规则 |
| Provider | `internal/provider/*` | Git、Registry、Kubernetes、DNS、NetworkPolicy 适配 |
| Worker | `internal/worker` | 异步任务运行器和调度 |
| 前端 | `web/src` | React 页面、API client、i18n、公共组件 |

## 三、核心流程概要

### 3.1 仓库构建流程

1. 用户连接 Git 账号并绑定仓库。
2. 用户创建部署配置，选择 Dockerfile、构建上下文、镜像站和目标 tag。
3. 手动触发、Webhook 或 Access Token 创建 BuildRun。
4. API 创建 BuildJob 并投递 Asynq 任务。
5. Worker 创建 Kubernetes Job，使用 BuildKit rootless 构建镜像。
6. Worker 推送镜像、记录 digest、更新 BuildRun/BuildJob。
7. 若部署配置启用自动部署，创建 Release 并投递部署任务。

### 3.2 镜像部署流程

1. 用户选择部署配置和镜像地址。
2. API 创建 Release。
3. Worker 生成 Kubernetes 资源规格。
4. Kubernetes Provider apply Deployment、Service、ConfigMap、Secret、PVC、Ingress。
5. Worker 等待 rollout，写入 ReleaseLog 和状态。

### 3.3 计费流程

1. Worker 定时采集运行资源和构建资源用量。
2. 根据费率表生成 BillingUsageRecord。
3. 根据 billed user 生成 BillingLedgerEntry。
4. 更新 UserWallet 余额。
5. UI 通过账单页查看余额、用量和账本。

## 五、设计约束

- 构建和部署长耗时任务必须进入 worker。
- 前端不得绕过后端直接编排外部平台 API。
- Secret 不回显，不以明文进入业务表。
- API 和 worker 使用同一数据库 schema，启动时执行迁移。
- docs 用户文档和 `Doc/devops` 工程文档职责分离。
