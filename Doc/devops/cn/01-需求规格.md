## 一、功能概述

Liteyuki DevOps 是面向个人开发者和小团队的应用交付平台。平台把代码仓库、镜像站、构建、Kubernetes 部署、访问入口、运行状态和计费记录统一到一个控制台中，让用户以项目空间和应用为核心组织交付流程。

第一阶段目标不是覆盖完整企业级 DevOps，而是稳定打通一条主链路：

```text
连接代码账号 -> 绑定仓库 -> 配置部署目标 -> 触发构建
  -> Worker 创建 Kubernetes Build Job -> 推送镜像
  -> 创建 Release -> 部署到运行集群 -> 暴露访问入口
```

## 二、功能需求清单

### 2.1 认证与准入

- 支持生产模式首个管理员初始化。
- 支持本地账号登录、退出、会话恢复和个人资料维护。
- 支持 OIDC Provider 后台配置、OIDC 登录、第三方身份绑定和解绑。
- 支持准入策略，包括邮箱域、邀请邮箱、OIDC 组和邮箱验证要求。
- 支持个人 Access Token，Token 只保存哈希，创建后不回显。

### 2.2 项目空间与应用

- 项目空间作为权限、资源归属、构建并发和计费归属边界。
- 应用归属于项目空间，表示一个可构建、可部署、可访问的服务。
- 应用基础信息仅维护名称、标识和图标；构建与运行参数由部署配置维护。
- 支持项目成员角色：owner、admin、developer、viewer。
- 支持项目固定、最近使用、使用频次和列表排序。

### 2.3 仓库与镜像站

- 支持 GitHub、Gitea 等 Git Provider 配置。
- 支持 Git OAuth 授权、仓库列表、分支、文件读取和 Webhook 创建。
- 支持 Harbor、DockerHub、Gitea Registry 等镜像站接入。
- 支持镜像站凭据、仓库搜索、tag 查询和镜像记录管理。
- Secret、Token、Registry Credential 不允许明文落业务表或回显到前端。

### 2.4 构建与发布

- 支持手动构建、Webhook 构建和 Access Token 触发构建。
- 构建以 BuildRun 表示业务运行，以 BuildJob 表示一次执行任务。
- Worker 使用 Asynq 消费任务，在目标运行集群创建一次性 Kubernetes Job。
- 构建执行器使用 BuildKit rootless，默认不挂载宿主机 Docker socket。
- 支持构建日志落库和 SSE 流式读取。
- 构建成功后可按部署配置自动创建 Release。

### 2.5 运行集群与部署

- 支持接入多个 Kubernetes/K3s 运行集群。
- 支持测试 kubeconfig、列出集群资源、查看 YAML、查看事件和删除资源。
- 部署配置维护副本数、CPU、内存、端口、环境变量、ConfigMap、Secret、数据卷和镜像来源。
- Release 生成 Deployment、Service、ConfigMap、Secret、PVC 和网关资源。
- 支持重启、回滚、运行日志、终端和数据导出。

### 2.6 访问入口与证书

- 支持 GatewayRoute 管理平台访问入口。
- 支持自动域名和自定义域名。
- 支持 Ingress/Traefik 路由资源生成。
- 支持基于 cert-manager ClusterIssuer 的 HTTP-01 证书申请。

### 2.7 应用市场

- 内置常见数据库、缓存、队列、对象存储、监控、开发工具和协作工具模板。
- 模板包含镜像、官网、官方仓库、默认资源、端口、参数、密钥和数据卷配置。
- 安装模板时创建应用、部署配置、密钥引用和可选首个 Release。

### 2.8 计费

- 余额、充值、补偿和扣费归属用户账户。
- 项目空间记录计费 owner；资源用量按项目、应用、资源和计量项生成 usage record。
- 账本记录用户余额变化，使用幂等键避免重复入账。
- 应用或项目删除不删除历史账单。

## 三、业务规则需求

### 3.1 命名与归属

| 对象 | 规则 |
| --- | --- |
| 项目空间 | slug 全局唯一，软删除项目不继续占用 slug |
| 应用 | slug 在同一项目空间内唯一，软删除应用不继续占用 slug |
| Kubernetes namespace | 默认使用 `ns-{projectIdShort}` |
| Deployment/Service | 使用 `dplt-{deploymentTargetIdShort}`，不依赖用户输入名称 |
| 计费 owner | 新资源按项目空间当前计费 owner 结算；所有余额变动归属用户 |

### 3.2 权限规则

- 后端负责最终权限判断，前端隐藏按钮只做体验优化。
- owner 不能被普通成员降级或删除；禁止删除最后一个 owner。
- 生产发布、Secret 写入、成员管理等操作必须校验角色。
- 危险操作必须写 AuditLog。

### 3.3 构建网络规则

- 构建 Job 默认不允许访问 Kubernetes API Server、元数据地址和未授权内网端口。
- DNS 53 端口应允许访问，以保证镜像解析和包管理可用。
- 私有镜像站、镜像源和非标端口通过白名单配置放行。

## 四、非功能需求

### 4.1 安全要求

- Secret 使用加密存储，只保存引用。
- Session 使用 HttpOnly Cookie；API 同时支持 Bearer Access Token。
- CORS 和 Cookie Origin Guard 必须开启。
- Webhook 必须校验签名，只接受已绑定仓库事件。
- 前端不直接调用 Git、镜像站、Kubernetes 等第三方平台 API。

### 4.2 可用性与可观测性

- API 提供 `/healthz` 健康检查。
- 构建、部署、Hook 和 Release 均需要状态、日志和错误信息。
- 构建日志支持断点续读。
- Worker 任务应具备超时、重试、心跳或可恢复机制。

### 4.3 性能与规模

- 所有潜在超过 100 条的列表 API 必须分页和排序。
- 前端列表统一使用 DataList 视觉和分页交互。
- 生产 API 内嵌前端静态文件，并对静态资源使用长缓存策略。

### 4.4 可维护性

- 后端采用模块化单体，Handler 只做参数解析和响应，业务逻辑进入 service/provider。
- 第三方系统访问由后端 provider 适配。
- 前端用户可见文本必须走 i18n。
- 新功能或逻辑改动必须同步更新文档站内容。
