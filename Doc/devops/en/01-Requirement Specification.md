## 1. Functional Overview

Liteyuki DevOps is an application delivery platform for individual developers and small teams. It connects source repositories, image registries, platform builds, Kubernetes deployments, gateway routes, runtime status, and billing records into one control plane.

The first-stage product focuses on one reliable delivery path:

```text
Connect Git account -> Bind repository -> Configure deployment target -> Trigger build
  -> Worker creates Kubernetes Build Job -> Push image
  -> Create Release -> Deploy to runtime cluster -> Expose route
```

## 2. Functional Requirements

### 2.1 Authentication and Admission

- Support production bootstrap for the first administrator.
- Support local login, logout, session resume, and profile updates.
- Support OIDC providers, OIDC login, and external identity binding.
- Support admission policies based on email domains, invite emails, OIDC groups, and verified email requirements.
- Support personal Access Tokens. Token plaintext is shown once and only token hashes are persisted.

### 2.2 Project Spaces and Applications

- Project spaces are boundaries for permissions, resource ownership, build concurrency, and billing ownership.
- Applications belong to project spaces and represent deployable services.
- Application metadata only keeps name, slug, and icon. Build and runtime settings belong to deployment targets.
- Project roles include owner, admin, developer, and viewer.
- Project lists support pinning, last-used ordering, use count, and pagination.

### 2.3 Repositories and Registries

- Support GitHub and Gitea provider configuration.
- Support Git OAuth, repository list, branch list, file read, and webhook setup.
- Support Harbor, DockerHub, Gitea Registry, and OCI-compatible registries.
- Support registry credentials, repository search, tag listing, and image records.
- Secrets, tokens, and registry credentials must not be stored or returned in plaintext.

### 2.4 Build and Release

- Support manual builds, webhook builds, and Access Token-triggered builds.
- BuildRun represents business execution. BuildJob represents one executor task.
- Worker consumes Asynq tasks and creates one-shot Kubernetes Jobs in runtime clusters.
- Build executor uses BuildKit rootless and never mounts the host Docker socket by default.
- Build logs are persisted and can be streamed through SSE.
- Successful builds may create Releases automatically based on deployment target policy.

### 2.5 Runtime Clusters and Deployment

- Support multiple Kubernetes/K3s runtime clusters.
- Support kubeconfig testing, resource listing, YAML inspection, event inspection, and resource deletion.
- Deployment targets define replicas, resources, service port, environment variables, config files, secret files, data volumes, and image source.
- Releases generate Deployment, Service, ConfigMap, Secret, PVC, and gateway resources.
- Support restart, rollback, runtime logs, terminal access, and data export.

### 2.6 Gateway and Certificates

- Support GatewayRoute management.
- Support generated domains and custom domains.
- Support Ingress/Traefik resource generation.
- Support HTTP-01 certificate issuance through cert-manager ClusterIssuer.

### 2.7 App Marketplace

- Provide built-in templates for databases, caches, queues, object storage, observability, developer tools, and collaboration tools.
- Templates include image, website, official repository, default resources, ports, parameters, secrets, and volumes.
- Installing a template creates an application, a deployment target, secret references, and an optional first Release.

### 2.8 Billing

- Wallet balance, recharge, compensation, and deductions belong to user accounts.
- Project spaces keep billing owner information.
- Usage records are generated by project, application, resource, and meter.
- Ledger entries record balance changes and use idempotency keys.
- Deleting projects or applications must not delete historical billing records.

## 3. Business Rules

| Object | Rule |
| --- | --- |
| Project space | Slug is globally unique among active projects |
| Application | Slug is unique within the project among active applications |
| Kubernetes namespace | Defaults to `ns-{projectIdShort}` |
| Deployment/Service | Uses `dplt-{deploymentTargetIdShort}` |
| Billing owner | New usage is charged to the current project billing owner |

## 4. Non-Functional Requirements

### 4.1 Security

- Secrets are encrypted and referenced by stable keys.
- Sessions use HttpOnly cookies. APIs also support Bearer Access Tokens.
- CORS and Cookie Origin Guard must be enabled.
- Webhooks must verify signatures and only accept bound repositories.
- Frontend must not call Git, registry, Kubernetes, or other third-party platform APIs directly.

### 4.2 Availability and Observability

- API exposes `/healthz`.
- Build, deploy, hook, and release flows must expose status, logs, and error messages.
- Build logs support resume by offset.
- Worker tasks should support timeout, retry, heartbeat, or recovery.

### 4.3 Performance and Scale

- List APIs that may exceed 100 items must support pagination and sorting.
- Frontend management lists should use the shared DataList pattern.
- Production API embeds the frontend SPA and serves hashed static assets with long cache headers.

### 4.4 Maintainability

- Backend uses modular monolith boundaries.
- Third-party integrations are adapted through backend providers.
- User-facing frontend text must use i18n.
- New features or behavior changes must update the documentation site.
