# 项目空间服务依赖与拓扑方案

## 1. 方案状态

- 状态：P1-P3 已实现，P4 观测验证保留为后续可选项
- 适用范围：同一项目空间内的应用与部署配置
- 首版目标：让用户能声明、查看和诊断“服务 A 依赖服务 B”这类关系，并在需要时为源部署配置注入稳定的集群内访问地址
- 设计原则：声明关系与 Kubernetes 实时资源分层，不把前端画线直接等同为运行时变更

## 2. 问题与目标

当一个项目空间内有 API、Web、Worker、数据库和消息队列等多个应用时，当前的应用拓扑只能回答“某个应用在 Kubernetes 中实际创建了什么”，无法回答：

- 哪个服务依赖哪个服务。
- 源服务应该使用哪个集群内地址访问目标服务。
- 修改、停用或删除目标服务会影响哪些应用。
- 当依赖失效时，问题是来自 Service、端口、Endpoint 还是跨集群。

本方案把这些能力拆成两层：

1. **逻辑关系层**：表达应用之间的依赖，可以由服务引用自动产生，也可以由用户手工记录。
2. **物理资源层**：继续使用现有应用拓扑接口，实时读取 Gateway、HTTPRoute、Service、Workload、Pod、ConfigMap、Secret 和 PVC。

项目空间拓扑展示逻辑关系；用户点开某个应用后，再展开它的 Kubernetes 物理拓扑。

## 3. 首版范围

### 3.1 实现内容

- 项目空间级拓扑页面。
- 明确到部署配置的服务引用。
- 仅用于说明关系的手工连接。
- 服务引用生成集群内主机名、端口或 HTTP URL，在源部署时注入环境变量。
- 目标 Service、端口、Endpoint 和集群一致性诊断。
- 依赖变更对重新发布的显式提示。
- 删除或停用被引用资源时的阻止与影响列表。
- 审计日志和平台事件。

### 3.2 不在首版实现

- 跨项目空间引用。
- 跨集群的 Kubernetes Service 直连。
- 自动把一个应用的 `prod` 映射到另一个应用的 `prod`；首版明确选择源和目标部署配置。
- Service Mesh、跨集群服务发现和调用链路编排。
- 从流量中自动推断并覆盖用户声明的依赖。
- 在引用地址中携带用户名、密码、Token 或其他 Secret。
- 根据拓扑自动创建或放宽 NetworkPolicy。
- 拖拽连线直接修改运行时。首版使用可校验的对话框创建关系。

## 4. 核心概念

### 4.1 服务引用 ServiceBinding

服务引用是一个会影响部署结果的配置。它明确指定：

- 哪个部署配置是调用方。
- 哪个部署配置是被调用方。
- 使用目标的哪个 Service 端口。
- 以什么环境变量形式注入地址。

服务引用不保存目标 Pod IP，只保存平台稳定 ID 和端口选择。每次发布源部署配置时，平台重新解析 Kubernetes Service 名称与命名空间。

### 4.2 手工拓扑关系 ProjectTopologyEdge

手工关系只用于说明系统设计，例如：

- Web 依赖 API，但地址由用户在其他配置中维护。
- Worker 消费某个消息服务。
- API 读写某个数据库应用。

它不注入环境变量、不创建 Kubernetes 资源，也不触发发布。

### 4.3 Kubernetes 物理拓扑

现有应用拓扑保持不变：

```text
Gateway -> HTTPRoute -> Service -> Deployment/StatefulSet -> Pod
                                      |-> ConfigMap
                                      |-> Secret
                                      |-> PVC
                                      |-> HPA
```

该层不持久化快照，每次查看时从集群即时计算。

## 5. 用户流程

### 5.1 创建可部署的服务引用

1. 用户进入项目空间的“拓扑”页。
2. 选择“添加关系”。
3. 关系模式选择“服务引用”。
4. 选择源应用和源部署配置。
5. 选择目标应用、目标部署配置和已声明的 Service 端口。
6. 选择注入方式并填写环境变量名。
7. 平台校验项目、集群、端口和环境变量冲突。
8. 保存后明确提示“需要重新发布源部署配置后生效”。
9. 用户确认后，复用现有 Release 能力创建一次发布；不确认则仅保存期望配置。

### 5.2 创建仅展示的手工关系

1. 关系模式选择“仅记录关系”。
2. 选择源应用和目标应用。
3. 选择关系类型：依赖、调用、读写、发布或消费。
4. 可选填写端口、协议和备注。
5. 保存后立即出现在拓扑中，不产生部署任务。

### 5.3 修改或删除被引用服务

当用户尝试删除应用或部署配置时，后端先查询活动服务引用：

- 无引用：继续原有删除流程。
- 有引用：返回稳定错误码 `service_binding_in_use`，并返回受影响的源应用和部署配置摘要。
- 用户需要先删除或停用引用，再删除目标资源。首版不提供级联删除。

手工拓扑关系不阻止删除，应用删除时可随应用一起清理。

## 6. 数据模型

### 6.1 `service_bindings`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | `sbind_` 前缀 ID |
| `project_id` | string | 所属项目空间 |
| `source_application_id` | string | 源应用 |
| `source_deployment_target_id` | string | 需要注入配置的源部署配置 |
| `target_application_id` | string | 目标应用 |
| `target_deployment_target_id` | string | 提供 Service 的目标部署配置 |
| `target_port_name` | string | 目标 Service 端口名，作为主要稳定引用 |
| `target_port` | int | 保存时的端口快照，用于展示和变更检测 |
| `protocol` | string | `http` / `https` / `tcp` |
| `path` | string | HTTP(S) 可选路径，默认空 |
| `injection_mode` | string | `url` / `host_port` |
| `url_env_var` | string | `url` 模式的环境变量名 |
| `host_env_var` | string | `host_port` 模式的主机环境变量名 |
| `port_env_var` | string | `host_port` 模式的端口环境变量名 |
| `enabled` | bool | 是否参与后续发布 |
| `created_by` | string | 创建人 |
| `created_at` / `updated_at` | time | 时间戳 |

约束：

- 源和目标必须属于 `project_id`。
- 源和目标部署配置不能相同。
- 建立联合唯一索引，防止同一源部署配置重复占用同一个注入变量。
- 运行时还要做服务层校验，不仅依赖数据库约束。
- 该表不保存密码、Token 或完整数据库 DSN。

### 6.2 `project_topology_edges`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | `ptedge_` 前缀 ID |
| `project_id` | string | 所属项目空间 |
| `source_application_id` | string | 源应用 |
| `source_deployment_target_id` | string nullable | 可选精确到部署配置 |
| `target_application_id` | string | 目标应用 |
| `target_deployment_target_id` | string nullable | 可选精确到部署配置 |
| `relation_type` | string | `depends_on` / `calls` / `reads_writes` / `publishes_to` / `consumes_from` |
| `protocol` | string | 可选协议 |
| `port` | int | 可选端口 |
| `description` | string | 最多 500 字符的非密钥备注 |
| `created_by` | string | 创建人 |
| `created_at` / `updated_at` | time | 时间戳 |

手工关系与服务引用使用两张表，避免把“展示关系”误当成“部署配置”。项目空间拓扑查询时再聚合两者，不额外持久化图快照。

## 7. 地址解析与部署语义

### 7.1 集群内地址

首版服务引用要求源和目标部署配置使用同一运行集群。目标主机名使用完整 Kubernetes Service DNS：

```text
<service-name>.<namespace>.svc.cluster.local
```

按当前资源命名规则，示例为：

```text
dplt-b530527f18.ns-c119e462fb.svc.cluster.local
```

资源名称必须由后端共享命名组件从 DeploymentTarget ID 解析，API 和 Worker 不能各自拼接。

### 7.2 环境变量注入

HTTP(S) `url` 模式：

```text
USER_SERVICE_URL=http://dplt-b530527f18.ns-c119e462fb.svc.cluster.local:8080/api
```

TCP 或数据库 `host_port` 模式：

```text
DATABASE_HOST=dplt-266715dfd4.ns-c119e462fb.svc.cluster.local
DATABASE_PORT=5432
```

用户名和密码继续由项目空间 Secret 或运行配置提供。平台不为数据库自动拼接带密码的 DSN。

### 7.3 变更生效方式

- 创建、修改、启用、停用或删除服务引用后，返回 `requiresRedeploy=true` 和受影响的源部署配置。
- 保存操作不默认立即重新发布，前端使用二次确认让用户选择“保存”或“保存并发布”。
- Worker 在生成源 Deployment/StatefulSet 规格前，由业务服务解析启用的 ServiceBinding，并将生成的变量作为一组结构化运行配置传入 Kubernetes provider。
- Kubernetes provider 不访问数据库，也不理解 ServiceBinding 业务模型。
- ServiceBinding 的归一化结果参与 Pod Template 指纹，确保引用变更后创建 Release 会稳定触发滚动更新。
- 目标应用只更新镜像、副本或 Pod 时，Service DNS 不变，不需要重新发布依赖方。
- 目标 Service 端口名被删除或改名时，引用变为 `invalid`，平台不在后台悄悄切换到其他端口。

## 8. API 设计

### 8.1 项目空间拓扑

```http
GET /api/v1/projects/:projectId/topology
```

查询参数：

- `stage`：可选，按部署阶段筛选。
- `origins`：可选，默认 `service_binding,manual`。
- `applicationId`：可选，仅返回与指定应用相关的上下游。

返回示例：

```json
{
  "generatedAt": "2026-07-18T10:00:00Z",
  "nodes": [
    {
      "id": "app_api",
      "kind": "application",
      "name": "API",
      "status": "healthy",
      "deploymentTargets": [
        {
          "id": "dplt_api_prod",
          "name": "prod",
          "stage": "prod",
          "clusterId": "cluster_main"
        }
      ]
    }
  ],
  "edges": [
    {
      "id": "sbind_user_service",
      "source": "app_web",
      "target": "app_api",
      "sourceDeploymentTargetId": "dplt_web_prod",
      "targetDeploymentTargetId": "dplt_api_prod",
      "origin": "service_binding",
      "relationType": "calls",
      "status": "ready",
      "protocol": "http",
      "port": 8080
    }
  ],
  "warnings": []
}
```

该接口不保存返回值快照。节点来自 Application/DeploymentTarget，连线来自 ServiceBinding 和 ProjectTopologyEdge。首版设置 500 个节点和 1000 条连线的硬上限，超出时返回 `topology_truncated` 警告，避免单次请求拖垮 API 和浏览器。

### 8.2 服务引用 CRUD

```http
GET    /api/v1/projects/:projectId/service-bindings
POST   /api/v1/projects/:projectId/service-bindings
PUT    /api/v1/projects/:projectId/service-bindings/:bindingId
DELETE /api/v1/projects/:projectId/service-bindings/:bindingId
POST   /api/v1/projects/:projectId/service-bindings/:bindingId/check
```

列表接口遵循统一分页格式：

```text
items/page/pageSize/sortBy/sortOrder/total/totalPages
```

`check` 只通过 Kubernetes API 检查 Service、端口、Endpoint 和 NetworkPolicy 摘要，不从 API 进程对用户提供的主机发起任意网络请求。

### 8.3 手工关系 CRUD

```http
GET    /api/v1/projects/:projectId/topology-edges
POST   /api/v1/projects/:projectId/topology-edges
PUT    /api/v1/projects/:projectId/topology-edges/:edgeId
DELETE /api/v1/projects/:projectId/topology-edges/:edgeId
```

列表接口同样分页；拓扑聚合接口不使用分页，而是使用过滤条件和图规模上限。

### 8.4 稳定错误码

- `service_binding_cross_project`
- `service_binding_cross_cluster`
- `service_binding_source_target_same`
- `service_binding_port_not_found`
- `service_binding_env_conflict`
- `service_binding_reserved_env`
- `service_binding_in_use`
- `service_binding_target_unavailable`
- `topology_edge_duplicate`
- `topology_truncated`

后端只返回稳定错误码和必要的原始诊断数据，用户可见文案由前端 i18n 映射。

## 9. 校验、权限与安全

### 9.1 权限

- Viewer：可查看项目拓扑和诊断结果。
- Developer：首版只查看，避免绕过部署配置管理边界。
- Project Owner/Admin：可管理服务引用和手工关系。
- Platform Admin：按现有项目空间权限旁路规则处理。

所有写操作都由后端重新校验源应用、目标应用和部署配置的 `project_id`，不信任前端级联选择器。

### 9.2 数据校验

- 环境变量名必须符合 `[A-Z_][A-Z0-9_]*`。
- 拒绝覆盖平台保留变量，也拒绝与部署配置现有环境变量重名。首版不做隐式覆盖。
- 协议仅允许 `http`、`https` 和 `tcp`。
- `url` 模式仅适用于 HTTP(S)，`host_port` 可用于全部协议。
- 目标端口必须来自 DeploymentTarget 已配置的 Service 端口列表，不允许自由输入任意端口。
- HTTP 路径只允许绝对路径，不允许 scheme、host、用户信息、query 或 fragment。
- 手工备注限长，按纯文本展示，不渲染用户提供的 HTML。
- 重复关系返回冲突错误；循环依赖允许保存，但在拓扑上显示 `dependency_cycle` 警告。

### 9.3 集群与网络边界

- 服务引用首版必须同集群。
- 跨集群时用户仍可创建“仅记录关系”，但不能生成 `.svc.cluster.local` 地址。
- 诊断检查目标 Service 是否存在、端口是否匹配、EndpointSlice 是否有 ready endpoint。
- 如存在 NetworkPolicy，首版只返回“可能受策略影响”的结构化警告，不自动放宽策略。
- 诊断不读取 Secret 值，不返回 Pod 环境变量的密钥内容。

## 10. 项目空间拓扑返回状态

逻辑连线的 `status` 使用稳定枚举：

| 状态 | 语义 |
| --- | --- |
| `ready` | 引用配置有效，目标 Service 与端口可解析 |
| `pending_release` | 引用已修改，源部署配置还没有通过新 Release 应用 |
| `unavailable` | 目标 Service 或 ready endpoint 不存在 |
| `invalid` | 部署配置、端口或集群关系不再满足约束 |
| `disabled` | 引用已停用，下次发布不注入 |
| `declared` | 手工拓扑关系，不做运行时可用性承诺 |

`pending_release` 需要可比较的引用版本。实现时为 ServiceBinding 生成规范化 digest，将该 digest 写入源 Workload annotation 或 Release 快照；当前 digest 与已部署 digest 不同时返回该状态。

## 11. 前端交互

### 11.1 项目空间拓扑页

在项目空间工作台新增“拓扑”项，不在平台侧边栏增加全局菜单。

页面结构：

- 顶部工具栏：阶段筛选、关系来源筛选、“适应画布”、刷新和“添加关系”。
- 主画布：ECharts graph，应用作为主节点，节点内展示健康状态和部署配置数量。
- 连线：服务引用使用实线，手工关系使用虚线；颜色只表达状态，不用来区分协议。
- 详情面板：点击节点展示应用与部署配置；点击连线展示来源、端口、注入变量、状态和诊断。
- 移动端：默认使用可搜索的依赖列表，用户按需切换到可缩放图形，避免小屏上强迫操作复杂画布。

### 11.2 创建关系对话框

复用 shadcn/ui、React Hook Form 和 Zod，使用渐进式字段：

- 基础区：关系模式、源应用、源部署配置、目标应用、目标部署配置。
- 服务引用区：目标端口、注入方式和环境变量名。
- 高级区：协议、HTTP 路径和备注。

应用和部署配置都使用项目现有通用选择器，不允许手填 ID、Service 名称或命名空间。

### 11.3 应用拓扑页

现有应用 Kubernetes 拓扑不与逻辑节点硬合并。页面增加“上游与下游”摘要，通过项目空间拓扑接口的 `applicationId` 筛选查询。用户可从摘要跳转到相关应用，物理 Kubernetes 资源仍由原接口单独刷新。

### 11.4 图形约定

- 自动生成的服务引用：实线。
- 手工记录的关系：虚线。
- 未来观测到的流量：仅作为第三层验证信号，不覆盖前两层；首版不实现。
- 线条动画只用于用户显式开启的实时观测模式，静态拓扑不持续动画。

## 12. 诊断结果

`POST /service-bindings/:bindingId/check` 返回结构化检查项：

| 检查项 | 示例 code |
| --- | --- |
| 源和目标属于同一集群 | `cluster_match` |
| 目标 Service 存在 | `service_exists` |
| 目标端口存在 | `service_port_resolved` |
| EndpointSlice 有 ready endpoint | `endpoint_ready` |
| 存在可能影响访问的 NetworkPolicy | `network_policy_detected` |
| 已部署引用 digest 与当前配置一致 | `binding_applied` |

每项包含 `status=passed|warning|failed`、稳定 `code` 和必要的资源摘要。前端根据 `code` 本地化，第三方/Kubernetes 原始 message 只放在可展开的详情中。

## 13. 事件与审计

写入 AuditLog 的操作：

- 创建、修改、启用、停用和删除服务引用。
- 创建、修改和删除手工拓扑关系。
- 因引用变更发起的发布。

平台事件：

- `service_binding.created`
- `service_binding.updated`
- `service_binding.deleted`
- `service_binding.invalid`
- `service_binding.recovered`

手工关系默认只写审计日志，不产生可通知事件，避免用户整理图布局时制造无意义通知。

## 14. 代码边界与实现顺序

### 14.1 后端边界

建议新增：

```text
internal/model/service_binding.go
internal/dependency/service.go
internal/dependency/repository.go
internal/dependency/resolver.go
internal/api/service_binding_handlers.go
internal/api/project_topology_handlers.go
internal/provider/kubernetes/service_dependency.go
```

职责：

- Handler：参数解析、项目权限入口和响应。
- Dependency service：校验关系、解析注入值、检查删除影响、聚合项目拓扑。
- Repository：ServiceBinding 和 ProjectTopologyEdge 的数据访问。
- Kubernetes provider：只读 Service、EndpointSlice 和 NetworkPolicy 摘要，不访问业务表。
- Worker：调用 dependency resolver 生成结构化环境变量，再传入现有 Kubernetes spec 组装流程。

需要把当前分散在 API 和 Worker 的 DeploymentTarget 资源命名规则收敛为共享纯函数，保证拓扑、Gateway、ServiceBinding 和部署生成使用同一名称。

### 14.2 前端边界

建议新增：

```text
web/src/pages/projects/project-topology-panel.tsx
web/src/pages/projects/project-topology-query.ts
web/src/pages/projects/service-binding-dialog.tsx
web/src/pages/projects/topology-edge-dialog.tsx
web/src/components/common/dependency-resource-select.tsx
```

ECharts 继续按需懒加载，拓扑查询使用 TanStack Query，表单使用 React Hook Form + Zod，所有用户可见文案使用 i18n。

### 14.3 数据库迁移

- 新增 `service_bindings` 和 `project_topology_edges` 表。
- 无需迁移现有 Kubernetes 拓扑，因为它本来就不持久化。
- 无需修改现有应用、部署配置和 Release 数据；没有新建引用的应用行为保持不变。
- 为项目、源/目标应用、源/目标部署配置和启用状态建立查询索引。
- Down migration 先删除外键与索引，再删除新表。

## 15. 分阶段交付

### P1：项目逻辑拓扑

- 实现 ProjectTopologyEdge 数据模型、迁移、CRUD 和权限。
- 实现项目空间拓扑聚合接口。
- 实现项目空间拓扑页、手工关系对话框和移动端列表。
- 这一阶段不修改部署运行时，可以先独立验收。

### P2：服务引用

- 实现 ServiceBinding 数据模型、CRUD 和完整校验。
- 解析同集群 Service DNS 和端口。
- 在发布规格中注入绑定环境变量并写入 digest。
- 实现“保存”/“保存并发布”确认流程。
- 项目拓扑合并服务引用连线。

### P3：诊断、保护和事件

- 实现 Service、端口、EndpointSlice 和 NetworkPolicy 检查。
- 实现被引用应用/部署配置的删除保护。
- 实现 `pending_release`、`invalid` 和 `unavailable` 状态。
- 接入 AuditLog、平台事件和通知规则。

### P4：观测验证（后续可选）

- 从网关流量、OpenTelemetry 或 Service Mesh 聚合观测到的调用关系。
- 只将观测结果作为“已发生流量”的证据，不自动修改 ServiceBinding 或手工关系。
- 为观测连线设置时间窗口和自动过期，不把高频流量样本保存到业务关系表。

## 16. 验收标准

### 16.1 后端

- Owner/Admin 可在同项目、同集群的两个部署配置间创建服务引用。
- Developer/Viewer 写入服务引用或手工关系时返回 403。
- 跨项目 ID 伪造、跨集群引用、自引用、端口不存在和环境变量冲突都返回稳定错误码。
- 源部署发布后，Pod Template 中存在期望的 URL 或 host/port 变量。
- 服务引用变更前不影响现有 Pod；创建新 Release 后触发滚动更新。
- 目标部署配置有活动引用时不能直接删除，响应包含受影响资源摘要。
- 手工关系的创建、修改和删除不产生 Release 或 Kubernetes 资源变更。
- 项目拓扑仅返回当前用户可见项目内的节点与关系。

### 16.2 前端

- 项目空间拓扑可按阶段和关系来源筛选。
- 实线与虚线能明确区分服务引用和手工关系。
- 节点与连线的状态使用统一语义状态组件，hover/focus 可查看详细诊断。
- 创建关系时只能从有权限的应用、部署配置和端口候选中选择。
- 服务引用保存后明确告知是否需要重新发布，不在后台静默重启应用。
- 移动端可以使用依赖列表完成查看、筛选和管理，不依赖精细拖拽。
- 所有用户可见文案均通过 i18n 键映射。

### 16.3 最小集成场景

1. 在同一项目空间和运行集群创建 `web-prod`、`api-prod` 和 `postgres-prod` 三个部署配置。
2. 建立 `web-prod -> api-prod`，注入 `API_URL`。
3. 建立 `api-prod -> postgres-prod`，注入 `DATABASE_HOST` 和 `DATABASE_PORT`，密码使用项目 Secret。
4. 分别发布源部署配置，检查 Pod Template 中的注入值。
5. 项目拓扑展示 `Web -> API -> PostgreSQL`，点开任一应用可继续查看它的 Kubernetes 物理拓扑。
6. 手工删除 `api-prod` Service，诊断结果变为 `unavailable`，但不在查询时自动重建 Service。
7. 重新发布 `api-prod` 恢复 Service，诊断与拓扑状态恢复。

## 17. 实现前最终确认项

下列决策已在本方案中给出默认结论，实现时如无新的产品要求，不再反复扩大范围：

- 首版服务引用精确到 DeploymentTarget，不做阶段自动映射。
- 服务引用只支持同项目、同集群。
- 保存引用不默认重新发布。
- 手工关系永远不改动运行时。
- 首版不保存观测流量连线。
- 密钥不进入 ServiceBinding，密钥只由现有 Secret 机制管理。

## 18. 实现结果

- 项目空间只在已有关系时展示“服务关系”标签；Owner/Admin 可从高级入口创建第一条关系，Developer/Viewer 不会看到空功能入口。
- 服务引用与手工关系分别持久化，手工关系不会进入发布规格；服务引用在 Worker 创建 Release 快照时统一解析并注入环境变量。
- ServiceBinding 规范化 digest 写入 Pod Template annotation。拓扑通过引用更新时间与最近成功 Release 的创建时间判断 `pending_release`。
- 诊断接口只读取 Kubernetes Service、EndpointSlice 和 NetworkPolicy，不拨测用户主机、不读取 Secret，也不自动修复被删除的 Service。
- 应用拓扑继续实时展示 Kubernetes 物理资源；存在逻辑关系时，应用页额外显示上游与下游摘要并可跳转项目空间拓扑。
- 创建、修改、删除、失效和恢复服务引用会进入平台事件；写操作同时记录 AuditLog。活动入向引用会阻止目标应用或部署配置被删除。
