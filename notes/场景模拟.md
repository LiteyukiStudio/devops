# 用户场景模拟

本文档用于在正式上线前模拟核心用户路径。当前阶段先确认场景是否合理；场景确认后，再按本文档逐项执行模拟验证，并把阻塞点回填到 `TODO.md`。每次场景测试发现的问题、修复动作和复测状态统一追加到 `docs/场景测试问题记录.md`。

## 验证原则

- 以真实用户路径为准，不只验证单个 API 或单个页面。
- 优先覆盖 MVP 基本功能：认证、项目空间、应用、部署配置、代码仓库、镜像站、构建、部署、访问、运行配置、权限和审计。
- 平台尚未上线，本地数据可清空重建；不为旧数据或旧路由保留兼容路径。
- 所有敏感值只验证“可写入、不可回显、可替换”，不在文档中记录真实 Secret。
- 场景验证时先跑主路径，再补失败路径和权限边界。
- 每次场景验证都必须同步维护 `docs/场景测试问题记录.md`，按批次记录问题、修复动作和复测结论。

## 通用前提

- 平台管理员已经完成首次初始化，并能登录后台。
- 至少存在一个可用 Kubernetes / K3s 集群，状态为 healthy。
- 至少存在一个可用镜像站，例如 Harbor、Gitea Registry 或 DockerHub。
- 至少存在一个 Git Provider，例如 GitHub 或 Gitea，并已配置 OAuth App 或允许 PAT 授权。
- Builder Agent 已注册，scope 能覆盖测试项目空间，标签能匹配默认构建配置。
- Worker 正常运行，Redis / PostgreSQL 可用。
- 测试用户包括：
  - `owner-user`：项目空间 Owner。
  - `dev-user`：项目空间 Developer。
  - `viewer-user`：项目空间 Viewer。

## 场景一：管理员完成平台基础配置

### 目标

验证平台管理员能完成全局基础资源配置，使普通用户后续可以创建应用并交付。

### 覆盖功能

登录、站点设置、OIDC Provider、Git Provider、镜像站、集群、Builder、系统状态展示。

### 操作路径

1. 管理员登录平台。
2. 设置站点标题、Logo、登录页副标题和默认语言。
3. 创建或确认 OIDC Provider，并保存准入策略。
4. 创建 Git Provider，填写实例地址、Client ID 和 Client Secret。
5. 创建全局镜像站，配置 endpoint、类型和默认凭据。
6. 接入 Kubernetes / K3s 集群，测试 kubeconfig。
7. 确认 Builder Agent 在线，scope 和标签可被项目空间使用。
8. 返回看板，查看集群、镜像站、构建器和近期状态。

### 期望结果

- 管理员能看到所有全局资源的健康状态。
- Client Secret、Git Token、Registry Credential 不回显明文。
- 看板可以展示集群、镜像站、Builder 的状态摘要。
- 普通用户不应在业务入口看到管理员维护的敏感连接配置。

## 场景二：个人开发者交付 monorepo 博客

### 目标

验证一个用户从代码仓库创建两个应用，并分别用 dev / prod 部署配置完成构建和部署。

### 用户故事

用户自己开发了一个博客系统https://github.com/snowykami/neo-blog，仓库是 monorepo，包含前端和后端。后端在仓库根目录，前端在 `web/` 目录下，两个应用都已经写好 Dockerfile。用户希望 dev 分支提交自动部署到开发环境，main 分支合并后自动部署到生产环境。

### 覆盖功能

项目空间、应用、仓库绑定、部署配置、Dockerfile 探测、构建上下文、分支触发、自动部署、发布记录、回滚、访问入口。

### 操作路径

1. `owner-user` 创建项目空间：`轻雪的博客`。
2. 在项目空间内创建两个环境：
   - `开发环境`，标识 `dev`。
   - `生产环境`，标识 `prod`。
3. 绑定 Git 凭据并选择博客仓库。
4. 创建应用 `blog-api`，图标选择 server。
5. 在 `blog-api` 下创建部署配置 `开发`：
   - 来源类型：代码仓库。
   - 仓库：博客仓库。
   - 分支规则：`dev`。
   - Dockerfile：`Dockerfile`。
   - 构建上下文：`.`。
   - 环境：`开发环境`。
   - 自动部署：开启。
6. 在 `blog-api` 下创建部署配置 `生产`：
   - 分支规则：`main`。
   - 环境：`生产环境`。
   - 自动部署：开启或需要人工确认，按产品当前能力选择。
7. 创建应用 `blog-web`。
8. 在 `blog-web` 下创建 dev / prod 两个部署配置：
   - Dockerfile：`web/Dockerfile`。
   - 构建上下文：`web`。
   - dev 匹配 `dev`，prod 匹配 `main`。
9. 手动触发一次 `blog-api` dev 构建，查看 BuildRun、BuildJob、日志和最终镜像。
10. 手动触发一次 `blog-web` dev 构建，确认目标镜像模板和镜像站凭据生效。
11. 模拟 dev 分支 webhook，确认平台只触发 dev 部署配置。
12. 模拟 main 分支 webhook，确认平台只触发 prod 部署配置。
13. 在部署页查看 Release 状态、部署日志和当前镜像。
14. 为前端 dev 部署配置创建默认域名访问入口。
15. 对 prod 发布执行一次回滚。

### 期望结果

- 一个 monorepo 通过多个应用表达多个可部署服务，而不是引入额外模块层级。
- BuildRun 直接绑定 `applicationId` 和 `deploymentTargetId`。
- dev / prod 构建和部署不会互相串线。
- 构建成功后能看到镜像引用、digest、触发人、Git 提交者和日志。
- 自动部署会生成 Release，发布失败时能看到业务化错误和日志。
- 访问入口直接选择部署配置，不要求用户单独选择环境和阶段。
- 回滚只允许回滚到同一应用、同一环境、同一部署配置的上一成功 Release。

## 场景三：只有镜像的第三方服务直接部署

### 目标

验证用户不绑定代码仓库，也能通过已有镜像创建应用并发布。

### 用户故事

团队需要在项目空间里运行一个第三方管理后台，例如 `nginx`、`adminer` 或内部已构建好的镜像。这个服务没有源码仓库，也不需要平台 Builder 构建。

### 覆盖功能

镜像站、镜像记录、应用、镜像来源部署配置、手动发布、运行配置、访问入口。

### 操作路径

1. `owner-user` 在已有项目空间中创建应用 `adminer`。
2. 创建部署配置 `开发`：
   - 来源类型：已有镜像。
   - 镜像引用：选择镜像站中的 `adminer:latest`，或手动填写完整镜像引用。
   - 环境：`开发环境`。
   - 自动部署：关闭。
3. 在部署页创建 Release，使用该镜像引用部署。
4. 查看 Release 运行日志和 Kubernetes 资源。
5. 创建访问入口，使用默认域名访问服务。
6. 修改镜像 tag 为新版本，再创建一次 Release。
7. 回滚到上一成功版本。

### 期望结果

- `sourceType=image` 的部署配置不进入 Builder 队列。
- 没有仓库绑定时，应用仍然可以完成发布、访问和回滚。
- Release 记录包含镜像引用、环境、部署配置和操作人。
- 集群资源归属展示为“项目空间 / 应用 / 部署配置”，不展示原始 ID 作为主要信息。

## 场景四：运行配置和密钥覆盖

### 目标

验证 Environment 默认配置与 DeploymentTarget 覆盖项能合并生成运行态 ConfigMap / Secret。

### 覆盖功能

环境、部署配置、ConfigMap、Secret、Secret 不回显、重新发布、集群资源查看。

### 操作路径

1. `owner-user` 在项目空间环境 `开发环境` 中配置默认运行参数：
   - `LOG_LEVEL=info`。
   - `FEATURE_FLAG=false`。
2. 在应用 `blog-api` 的 dev 部署配置中添加覆盖项：
   - ConfigMap：`FEATURE_FLAG=true`。
   - Secret：`DATABASE_URL=<测试数据库连接串>`。
3. 保存后重新打开编辑页，确认 Secret 明文不回显。
4. 创建或重新发布 dev Release。
5. 在集群资源页面查看该部署配置生成的 ConfigMap、Secret、Deployment 和 Service。
6. 修改 Secret 值为空保存，确认含义是“不修改已有 Secret”。
7. 输入新的 Secret 值保存，再发布一次。

### 期望结果

- 同名普通配置以部署配置覆盖项为准。
- Secret 只允许写入，不在 API 和前端回显明文。
- 发布时 Worker 使用“环境默认 + 部署配置覆盖”生成运行态资源。
- 集群资源能按项目空间、应用和部署配置筛选或识别。

## 场景五：项目协作与权限边界

### 目标

验证 Owner、Developer、Viewer 的权限边界，以及危险操作审计。

### 覆盖功能

成员管理、角色权限、按钮显隐、后端权限校验、AuditLog、危险操作确认。

### 操作路径

1. `owner-user` 邀请 `dev-user` 加入项目空间，角色为 Developer。
2. `owner-user` 邀请 `viewer-user` 加入项目空间，角色为 Viewer。
3. `dev-user` 登录后尝试：
   - 创建应用。
   - 创建部署配置。
   - 手动触发 dev 构建。
   - 部署 dev Release。
   - 删除应用。
   - 管理项目级镜像站。
4. `viewer-user` 登录后尝试：
   - 查看项目空间、应用、构建和发布。
   - 触发构建。
   - 修改部署配置。
   - 查看 Secret。
5. `owner-user` 尝试删除应用，确认需要二次确认。
6. 管理员查看 AuditLog。

### 期望结果

- Developer 可以创建应用、维护交付配置、构建和部署非生产环境。
- Developer 不可以删除应用或管理项目级镜像站。
- Viewer 只能查看，不可以触发构建、部署或写入配置。
- 前端隐藏无权限操作，但后端仍必须最终拒绝越权请求。
- 删除、成员变更、Secret 写入、Token 操作等危险操作必须产生审计记录。

## 场景六：Access Token 触发自动化构建

### 目标

验证外部系统可以使用平台 Access Token 触发构建，同时受 scope 和项目权限限制。

### 覆盖功能

Access Token、scope、API 触发构建、BuildRun 记录、Token 撤销。

### 操作路径

1. `owner-user` 在账号页创建 Access Token，只选择构建触发所需 scope。
2. 保存 Token，确认 Token 只展示一次。
3. 使用该 Token 调用构建触发 API，指定项目空间、应用和部署配置。
4. 查看应用构建页，确认出现 triggerType=`api` 的 BuildRun。
5. 尝试用同一个 Token 调用无 scope 的 API，例如删除应用或管理成员。
6. 撤销 Token。
7. 再次调用构建触发 API。

### 期望结果

- Token 只保存 hash，不再回显原文。
- 有 scope 的构建触发可以成功进入队列。
- 无 scope 的 API 默认拒绝。
- 撤销后的 Token 不能继续使用。
- BuildRun 能展示 API 触发来源，不冒充 Git 提交者。

## 场景七：失败诊断和恢复

### 目标

验证平台在常见失败情况下能给出可理解的状态、日志和恢复入口。

### 覆盖功能

构建失败、镜像推送失败、部署失败、网关失败、日志、重试、回滚。

### 操作路径

1. 创建一个 Dockerfile 路径错误的部署配置，触发构建。
2. 查看 BuildRun 状态、BuildJob 日志和错误信息。
3. 修正 Dockerfile 路径，重试构建。
4. 临时配置错误镜像站凭据，触发构建推送失败。
5. 恢复镜像站凭据，重试构建。
6. 创建一个服务端口错误的部署配置并发布，观察访问或 rollout 失败。
7. 修正部署配置服务端口或镜像配置后重新发布。
8. 创建一个 DNS/CNAME 未验证的自定义域名，查看访问入口状态。

### 期望结果

- 构建失败、部署失败、网关失败都能在对应页面看到状态 Badge 和日志入口。
- 失败信息不直接暴露底层堆栈或 Secret。
- 终态构建可以重试；成功历史发布可以用于回滚。
- 网关和证书状态能独立展示，避免用户误以为应用部署失败。

## 场景确认后建议的模拟顺序

1. 先跑场景一，确认平台基础资源可用。
2. 再跑场景二，验证源码构建到部署的主闭环。
3. 再跑场景三，验证无源码镜像直部署。
4. 再跑场景四，验证运行配置和 Secret。
5. 再跑场景五和场景六，验证权限和外部 API。
6. 最后跑场景七，验证失败路径和恢复体验。

## 待确认问题

- prod 环境是否默认要求人工确认，还是 MVP 中允许自动部署。
- PR Merge 事件是否在当前阶段直接等价为 main 分支 push webhook。
- 自定义域名验证是否要求真实 DNS，还是本地模拟只验证 CNAME 提示和状态流转。
- 本地场景验证是否允许清空数据库和重新部署 dev compose。

   - Secret 列表只展示 `data hidden`，没有回显密钥内容。

### 阻塞和体验卡点

1. 新应用创建失败
   - 操作：在项目空间应用 tab 中创建 `场景测试应用 / scenario-app`。
   - 结果：提交后 toast 报错：
     `ERROR: null value in column "source_type" of relation "applications" violates not-null constraint (SQLSTATE 23502)`。
   - 判断：应用最小单元重构后，前端创建应用只提交基础信息，但数据库或模型里仍残留旧 `applications.source_type` 非空约束。
   - 影响：场景二、场景三无法从 UI 新建应用，主路径被阻断。

2. 应用详情缺少创建部署配置入口
   - 操作：进入已有应用详情页，检查概览、构建、部署、访问 tab 的可用动作。
   - 结果：页面只出现“保存配置”“触发构建”“创建发布”“创建域名”，没有“创建部署配置 / 新增部署配置”入口。
   - 判断：旧模块页签移除后，新的 DeploymentTarget 创建和编辑入口没有补齐到应用详情工作流。
   - 影响：即使应用创建成功，用户也无法通过 UI 完成“应用 -> 部署配置 -> 构建/部署”的核心配置。

3. 现有部署配置数据不完整，导致触发构建禁用
   - 现象：已有应用的构建 tab 中“触发构建”按钮为 disabled。
   - 数据盘点：当前 `deployment_targets` 存在记录，但 `repository_binding_id`、`build_provider_id`、`target_registry_id`、`target_repository` 等关键字段为空。
   - 判断：本地历史数据来自旧模型，无法满足新模型触发构建所需配置。
   - 影响：无法用现有应用跑新 BuildRun，也无法验证自动部署链路。

4. 历史 BuildRun / Release 无部署配置归属
   - 现象：应用概览显示存在成功发布记录，但部署健康显示 `0 / 1 个部署成功`；部署 tab 中部署配置显示“未部署”。
   - 数据盘点：历史 `build_runs.deployment_target_id` 和 `releases.deployment_target_id` 为空。
   - 影响：新部署页无法把历史构建和发布归属到部署配置，行内“部署”按钮禁用。
   - 说明：这符合“不兼容旧数据”的原则，但本地若要继续完整验收，需要清库重建或补一组新模型数据。

5. 顶部“创建发布”和行内“部署”的状态不一致
   - 部署列表行内“部署”按钮禁用。
   - 顶部“创建发布”按钮仍可打开弹窗，并列出旧构建记录。
   - 选择旧构建后能带出镜像，但部署配置字段仍为 `-`，保存保持 disabled。
   - 体验问题：用户容易误以为可以从顶部创建发布，但最终不知道为什么不能保存。

6. 创建应用弹窗描述仍是旧口径
   - 弹窗描述为“先支持 repository 和 image 两类来源，部署和构建后续联调。”
   - 当前产品口径是“应用只维护基础信息；repository / image 来源归属部署配置”。
   - 影响：新用户会误解应用创建阶段需要选择来源类型。

7. 旧 Kubernetes 资源仍显示部署配置为 `-`
