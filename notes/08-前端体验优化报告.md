# 前端体验与表单复杂度优化报告

## 背景

本次走查以“部署上线一个 Web 项目”为主路径，先启动本地开发依赖、API 和前端，再用浏览器实际登录控制台，查看项目空间、应用详情、部署配置、创建部署配置弹窗，以及移动端视口表现。

验证环境：

- `docker compose -f docker-compose-dev.yaml up -d --build`
- `ENV_FILE=.env.development go run ./cmd/api`
- `pnpm --dir web dev --host 127.0.0.1`
- 前端地址：`http://127.0.0.1:5173`

## 总体结论

平台能力已经比较完整，但关键路径仍然偏“资源配置后台”，不像“帮用户把项目部署上线”的产品。最大问题不在单个控件，而在信息出现顺序：用户还没完成第一条部署，就已经被构建规格、运行规格、发布策略、数据卷、ConfigMap、Secret、文件挂载等概念同时包围。

建议把体验目标改成：

> 先跑起来，再慢慢把交付变轻松。

普通 Web 项目的首条部署，用户只应该做少量决策：选择代码或镜像、选择目标镜像站、确认端口、保存并触发构建。CPU、内存、副本、构建超时、数据卷、配置文件、密钥文件、并发策略都应作为默认值或高级配置出现。

## 主要问题

### P0：新增部署配置弹窗信息密度过高

位置：`web/src/pages/applications/application-deployments-panel.tsx`

实际弹窗一次性展示了约 29 个 label 和 30 多个输入控件，包含：

- 名称、阶段、集群
- 来源类型、仓库、镜像站、Dockerfile、构建上下文、构建路径、镜像引用
- 构建 CPU、构建内存、构建超时
- 服务端口、副本数、CPU、内存
- 分支匹配、标签匹配、并发策略、自动部署
- 运行数据、数据卷
- 公共配置、普通配置、ConfigMap 文件、密钥配置、密钥文件

这会让新用户误以为“所有项都必须懂”。尤其是 1c/1Gi/1 副本、8080 端口、Dockerfile、构建上下文这些已经有默认值的内容，不应该占据同等权重。

建议：

- 弹窗改为分组折叠，默认只展开“基础部署”。
- 默认展示摘要，不直接展示可编辑字段：`生产 · 1 副本 · 1c · 1Gi · http:8080 · 自动部署开启`。
- “构建设置”“运行规格”“发布策略”“运行数据”“运行配置与密钥”默认折叠。
- 数据卷只有在“保留运行数据”开启后再显示编辑器。

### P0：部署列表列数过多，错误信息抢占视线

位置：`web/src/pages/applications/application-deployment-targets-list.tsx`

当前部署列表有 15 列：名称、部署配置、阶段、运行规格、运行数据、自动部署、运行状态、资源占用、内网访问、版本、目标镜像、发布状态、部署进度、时间、操作。

移动端下表格内部横向宽度达到约 1757px；失败信息会以较长文本出现，即使有截断，也会让整行视觉重心偏向错误详情。

建议：

- 首屏列保留：名称、阶段、运行规格、运行状态、发布状态、镜像摘要、操作。
- 内网访问、资源占用、版本、完整错误、运行数据放入展开详情。
- 错误列只显示状态摘要，例如“集群连接失败”，完整错误放 hover/copy 或详情抽屉。
- 移动端改为卡片行，不使用宽表格作为主体验。

### P1：应用详情默认进入“概览”，但底部出现编辑表单

位置：`web/src/pages/applications/ApplicationConfigPage.tsx`

应用详情概览页本来用于判断当前应用是否可交付，但页面下方直接出现“应用图标、应用名称、应用标识、保存配置”。这会把“查看状态”和“编辑应用元信息”混在一起。

建议：

- 概览只展示状态和关键入口。
- 应用元信息编辑移动到“设置”或单独的“编辑应用”按钮。
- 保存按钮只在设置表单页面出现，避免用户以为看概览也需要保存。

### P1：运行配置存在两套心智模型

位置：

- `web/src/pages/applications/application-deployments-panel.tsx`
- `web/src/pages/applications/application-runtime-config-set-dialog.tsx`

部署配置里既有“公共配置”选择，又有内联环境变量、ConfigMap 文件、Secret 文本、Secret 文件。能力完整，但对新用户来说很难判断该用哪一种。

建议：

- 快速部署里只保留“选择配置集”。
- 内联环境变量和文件挂载归入“高级覆盖”折叠区。
- 文案统一为“配置集是推荐方式；高级覆盖只作用于当前部署配置”。

### P1：镜像站凭据表单把高级模板暴露得太早

位置：`web/src/pages/registries/registry-dialogs.tsx`

创建凭据时要求用户面对 `repositoryTemplate`、`tagTemplate`。这类字段是平台命名策略，不是连接镜像站的必要心智。

建议：

- 基础区只保留：镜像站、凭据名称、用途、用户名、密码或 Token。
- 仓库模板、Tag 模板放入“镜像命名规则”高级区。
- 如果系统已有推荐模板，默认填好并用摘要展示。

### P1：代码源 Provider 配置偏平台管理员视角

位置：`web/src/pages/code-repositories/code-repositories-dialogs.tsx`

Provider 弹窗把名称、类型、Base URL、作用域、认证方式、Client ID、Client Secret、OAuth 创建指南放在同一个表单。对于只想绑定 GitHub 仓库的用户，这一步显得过重。

建议：

- 用户侧默认走“连接 GitHub / Gitea 账号”。
- Provider 管理放到系统设置或高级入口。
- OAuth 应用创建指南可以保留，但应作为可展开帮助，不打断主表单。

### P2：发布弹窗仍要求用户理解构建运行和镜像引用

位置：`web/src/pages/applications/application-create-release-dialog.tsx`

创建发布需要选择构建运行、部署配置、镜像。对新用户来说，这些字段看起来像重复确认。

建议：

- 从部署配置行触发发布时，默认锁定部署配置。
- 默认选择最近成功构建。
- 镜像引用作为只读摘要展示，需要覆盖时再展开编辑。

### P2：本地开发构建上下文偏大

本次 `docker compose -f docker-compose-dev.yaml up -d --build` 观察到构建上下文传输较大。虽然这不是用户控制台体验，但会影响开发反馈速度。

建议：

- 检查根目录 `.dockerignore` 是否还需要排除文档截图、临时资产、测试输出和本地缓存。
- 尽量让 API/worker Dockerfile 只接收后端构建所需文件。

## 推荐重组方案

### 部署配置弹窗

默认展开：

1. 基础部署
   - 名称
   - 阶段
   - 来源类型
   - 仓库或镜像
   - 目标镜像站
   - 服务端口

默认折叠：

- 构建设置：Dockerfile、构建上下文、构建目录、构建资源、超时。
- 运行规格：副本数、CPU、内存，默认 `1 副本 / 1c / 1Gi`。
- 发布策略：分支匹配、标签匹配、并发策略、自动部署。
- 运行数据：保留数据、数据卷。
- 配置与密钥：配置集、环境变量覆盖、文件挂载、Secret。

### 部署列表

主行只回答三个问题：

- 这个部署配置是谁：名称、阶段、规格。
- 当前能不能跑：运行状态、发布状态。
- 下一步做什么：构建、发布、日志、编辑。

其他信息进入详情行或抽屉：

- 完整镜像引用
- 内网访问地址
- 资源指标
- 版本号
- 完整错误信息
- 数据卷与配置集

### 前置资源

代码库、镜像站、集群资源应按任务组织：

- “我想部署 GitHub 项目”：连接账号、选择仓库。
- “我想推送镜像”：选择镜像站、填写凭据。
- “我想部署到集群”：选择默认集群，失败时再进入集群配置。

不要让用户在第一条部署前先理解所有平台资源模型。

## 分阶段落地

### 落地记录

- 2026-06-30：第一阶段低风险减负已先落地部署配置弹窗。新增部署配置默认只展开“基础部署”，构建设置、运行规格、发布策略、运行数据、配置与密钥改为渐进式折叠区并展示摘要；Dockerfile、构建上下文、目标镜像和构建资源从来源字段拆出；未启用保留运行数据时不再展示数据卷编辑器。
- 2026-06-30：按模块并行完成报告剩余体验项。部署列表减为核心列，完整错误、内网访问、资源占用、版本和运行数据进入详情，移动端改为卡片；发布弹窗默认锁定部署配置并将镜像覆盖收进高级区；应用概览新增“部署上线”任务式入口，并将应用元信息编辑移到设置 tab；镜像站凭据、代码源 Provider 的高级字段改为渐进式折叠区；根目录 `.dockerignore` 收紧本地缓存、测试输出、文档站产物和临时资产，减少 Docker build context。

### 第一阶段：低风险减负

- [x] 部署配置弹窗增加折叠分组，默认只展开基础部署。
- [x] 数据卷编辑器在禁用保留数据时隐藏。
- [x] 运行规格改成摘要行，点击“调整”后展开。
- [x] 部署列表减少默认列，把完整错误移入详情。
- [x] 镜像站凭据的模板字段移入高级区。

验收标准：

- [x] 普通 Dockerfile Web 项目创建部署配置时，首屏必填决策不超过 6 个。
- [x] 默认 `1 副本 / 1c / 1Gi / http:8080` 可见但不打扰填写。
- [x] 移动端没有必须横向滚动才能完成创建的表单。

### 第二阶段：任务式部署向导

- [x] 在应用详情增加“部署上线”主按钮。
- [x] 向导拆为：选择来源 -> 确认构建 -> 确认运行 -> 发布。
- [x] 缺少仓库、镜像站、集群时，用前置检查卡片引导补齐。
- [x] 创建完成后直接提供“触发构建”或“查看部署进度”。

验收标准：

- [x] 新用户从项目空间进入，到触发第一次构建，不需要离开当前任务流。
- [x] 每一步只解释当前必须理解的概念。

### 第三阶段：统一表单体验

- [x] 抽象通用 `AdvancedSection` 或 `ProgressiveFieldset`。
- [x] 折叠区显示稳定摘要，并记住用户展开状态。
- [x] 对表单字段按“基础、来源、构建、运行、访问、数据、配置、安全”统一排序。
- [x] 所有复杂字段提供示例、默认值和“什么时候需要改”的一句话说明。

## 一句话方向

Luna DevOps 不应该让用户“配置一个 Kubernetes 发布系统”，而应该让用户感觉“选好代码，确认默认值，然后它帮我跑起来”。高级能力当然要有，但应该在用户需要时才出现。
