# Luna DevOps 前端视觉体系后续整改方案

本文记录 Luna DevOps 完成第一轮视觉体系整改后的剩余问题与后续实施方案。已完成的设计 token、页面模板、公共语义布局、项目空间列表样板、看板重构和全局设置宽度收口等事项不再重复列为待办。

本轮结论来自 2026-07-23 对生产控制台的浏览器巡检，覆盖桌面端主要导航页面、项目空间与应用详情，以及 390px 移动端关键页面。本文只保留仍需处理或需要继续收口的事项。

## 1. 当前状态

第一轮整改已取得以下结果：

- 资源列表、概览、设置和工具工作区已经形成基本页面类型边界。
- 项目空间列表的搜索、筛选、排序、主操作与 `DataList` 已基本收口。
- 看板已经增加重点关注、指标组、近期活动和平台准备度分层。
- 项目空间概览已经具备健康指标、最近构建、运营摘要和事件分区。
- 应用概览已经形成部署步骤、健康指标、运行信息和部署摘要。
- 全局设置内容宽度和保存操作位置较旧版明显改善。
- 状态 Badge、列表操作、分页、主题与响应式基础能力保持一致。
- 桌面端和 390px 移动端未发现页面级整体横向溢出。
- 项目级开发规范已经加入页面类型、语义布局、状态色、间距与 `DataList` 外壳约束。

当前前端已经从“通用管理后台”进入相对成熟的产品化阶段。后续重点不再是继续增加基础抽象，而是收口加载、空白、失败、异常优先级和移动端任务编排等真实使用状态。

## 2. 后续目标与边界

### 2.1 目标

- 让路由切换、加载、空状态和外部工具失败状态具有正式产品完成度。
- 消除侧边栏、开发悬浮工具和移动端表格中的遮挡与布局噪声。
- 让构建、发布、入口等异常在概览首屏可直接识别并进入处理流程。
- 减少应用概览中的等权卡片，强化主任务和下一步行动。
- 将移动端从简单纵向堆叠调整为适合小屏的筛选、操作和信息优先级。
- 统一剩余设置页、主操作、技术词汇与品牌主题色使用强度。

### 2.2 非目标

- 不重新设计已经稳定的 `DataList`、`ContentTabs`、状态 Badge 和表单基础控件。
- 不为填补空白增加无业务价值的装饰卡片。
- 不改变后端 API、权限规则、计费逻辑和构建部署业务语义。
- 不在本轮引入新 UI 框架或大规模视觉依赖。
- 不强制终端、日志和第三方品牌图标跟随用户主题色。

## 3. P0：必须优先修复

### 3.1 统一页面加载体验

#### 现状

路由切换和首次进入页面时存在多种低完成度加载形式：

- 整页空白，仅在中央显示“加载中…”。
- 页面框架已经出现，正文用一个大边框容器包裹“加载中…”。
- 看板、列表和应用市场加载时没有保留最终布局轮廓。
- 页面切换时内容闪烁，用户无法判断将要出现的结构。

#### 整改

新增或完善按页面类型复用的加载组件：

```text
AppLoadingState        首次会话和公共配置初始化
DataListSkeleton       表头、4 至 6 行骨架、分页占位
OverviewSkeleton       关注区、指标组、主次内容区骨架
SettingsSkeleton       tab、标签与字段骨架
TemplateGridSkeleton   应用市场工具栏与模板卡骨架
ToolViewportSkeleton   iframe、日志、拓扑和终端工作区骨架
```

要求：

- AppLayout 首次初始化允许使用全屏品牌加载态。
- 路由切换时保留侧边栏和顶栏，只替换正文区域。
- Skeleton 的宽度、高度和最终内容接近，避免明显布局跳动。
- 不再以普通 Card 单独包裹一行“加载中…”。
- 弱网状态下仍显示当前页面标题与基本操作上下文。
- 加载文案必须使用 i18n；骨架本身使用合适的 `aria-busy` 或状态说明。

#### 涉及文件

- `web/src/layouts/AppLayout.tsx`
- `web/src/pages/dashboard/DashboardPage.tsx`
- `web/src/pages/events/EventsPage.tsx`
- `web/src/pages/app-templates/AppTemplatesPage.tsx`
- `web/src/components/common/data-list.tsx`
- 新增的公共 skeleton 组件

#### 验收

- 人为限制网络后依次进入看板、项目空间、事件和应用市场。
- 加载期间无整页空白、无大白框单行文案、无明显布局跳动。
- 内容加载完成后 skeleton 完整消失。

### 3.2 修复侧边栏 Footer 遮挡导航

#### 现状

在桌面端中等高度视口下，侧边栏底部主题切换和用户面板覆盖“个人”分组，账号导航只显示一部分，账单入口可能完全不可见。

#### 整改

- 侧边栏使用明确的 `header + min-h-0 scrollable content + shrink-0 footer` 布局。
- Footer 不通过覆盖式定位压住导航内容。
- `SidebarContent` 底部不依赖猜测的固定 padding 补偿 Footer。
- 导航内容较长时只滚动 Content，品牌头和 Footer 保持稳定。
- 分组分隔线、滚动条和 Footer 之间保留合理安全距离。

#### 涉及文件

- `web/src/layouts/AppLayout.tsx`
- `web/src/components/ui/sidebar.tsx`
- `web/src/components/common/sidebar-user-panel.tsx`

#### 验收

分别使用 768px、844px、900px 和 962px 视口高度检查：

- 个人分组的账号和账单均可通过滚动访问。
- Footer 不覆盖任何导航文字或点击区域。
- 移动端 Sheet 内采用相同结构。

### 3.3 修复空列表分页语义

#### 现状

无数据时部分列表显示：

```text
第 1 / 0 页，共 0 条
```

同时仍展示每页条数、当前页和上一页/下一页。

#### 整改

- `total === 0` 时不展示页码、每页条数和翻页器。
- 空状态只显示“共 0 条”或完全隐藏分页摘要，由调用页面选择。
- API 偶发返回 `page=1,totalPages=0` 时由公共分页组件规范化展示，不要求每个页面自行修复。
- `total > 0` 时继续使用现有分页交互。
- 为 `PaginationController` 和 `DataList` 增加零数据测试。

#### 涉及文件

- `web/src/components/common/pagination.tsx`
- `web/src/components/common/data-list.tsx`
- 对应测试文件

### 3.4 避免开发悬浮面板遮挡业务内容

#### 现状

Debug 悬浮按钮在桌面和移动端会覆盖表格操作列、账单数字、表单字段和应用详情卡片，影响真实视觉验收。

#### 整改

- 默认停靠于视口安全边缘，四周至少保留 16px。
- 移动端缩小触发器，或默认收进侧边栏/开发工具菜单。
- 避让 `Dialog`、`Sheet`、sticky footer、表格固定操作列和 toast 区域。
- 增加开发环境视觉验收隐藏开关，并持久化到本地。
- 再次确认生产模式不渲染 Debug 面板；若生产部署仍显示，必须修正环境判断。

#### 涉及文件

- `web/src/components/common/debug-floating-panel.tsx`
- `web/src/layouts/AppLayout.tsx`

### 3.5 移动端事件筛选改为渐进披露

#### 现状

390px 下事件页首屏被查看范围、项目空间、应用、部署配置、分类、类型、级别、结果和日期范围完全占据，用户无法直接看到事件记录。

#### 整改

移动端结构调整为：

```text
搜索框
[筛选 n] [刷新]
已启用筛选 chips
事件列表
```

- 搜索框、筛选按钮和刷新按钮留在主页面。
- 其余筛选条件进入右侧或底部 Sheet。
- 筛选按钮显示已启用条件数量。
- 已启用筛选在列表上方显示为可移除 chips。
- Sheet 内提供“清除全部”和“应用筛选”。
- 桌面端保留多列筛选布局，但可评估将低频筛选放入“更多筛选”。
- 筛选条件、URL 参数和现有查询行为保持不变。

#### 涉及文件

- `web/src/pages/events/EventsPage.tsx`
- 可新增事件筛选 Sheet 私有组件
- `web/src/pages/events/event-filter-defaults.ts`

#### 验收

- 390px 首屏可看到至少一条事件记录或空状态标题。
- 打开、应用、清除筛选后 URL 与查询结果正确。
- Sheet 关闭后不丢失已应用筛选。

### 3.6 概览指标补齐风险语义

#### 现状

项目和应用概览已经使用指标组，但异常数据仍主要以中性黑色数值展示。例如：

- 最近构建失败。
- 发布记录中失败占比较高。
- 访问入口存在但 0 个可用。
- 应用部署失败，而指标卡仅在角落显示 Badge。

#### 整改

- `MetricItem` 支持 `neutral/success/warning/danger`，并由业务状态显式传入。
- 异常指标使用轻量语义背景、边框或数值强调，不使用大面积实色。
- 零值正常指标弱化；零值异常指标必须按危险语义展示。
- “总发布数”不是健康指标，优先展示失败数、成功率或“成功/总数”。
- “入口总数”应与“可用数”组合展示，例如 `0 / 6 可用`。
- 可点击指标进入带对应筛选的页面。
- 色彩之外必须保留明确文字，不能只靠红绿区分。

#### 涉及文件

- `web/src/components/common/metric-group.tsx` 或现有等价组件
- `web/src/pages/dashboard/DashboardPage.tsx`
- `web/src/pages/projects/ProjectWorkspacePage.tsx`
- `web/src/pages/applications/application-overview-panel.tsx`
- `web/src/pages/billing/billing-summary.tsx`

## 4. P1：主要视觉收口

### 4.1 建立两类正式空状态

#### 可行动空状态

适用于代码库、镜像站、集群、访问入口、通知渠道等首次配置场景。

内容：

- 语义图标或轻量品牌图形。
- 明确标题。
- 一句话说明用户将获得什么能力。
- 权限允许时提供主操作。
- 必要时提供文档入口。

#### 过滤结果为空

适用于搜索或筛选后没有结果。

内容保持紧凑：

- “没有符合当前条件的结果”。
- 当前条件摘要。
- “清除筛选”操作。

#### 规则

- 空状态不继续展示无意义分页。
- 不为填补空间添加无业务价值卡片。
- 权限不足时说明原因，不展示不可执行的主按钮。
- 不同资源可以使用不同图标，但布局和排版复用公共 `EmptyState`。

#### 涉及页面

- 代码库
- 镜像站
- 集群资源
- 应用访问入口
- 通知
- 项目空间和其他 DataList 页面

### 4.2 继续减少应用概览等权卡片

#### 现状

应用概览仍包含部署上线主卡、五个步骤块、四个指标卡、运行概览、访问入口、部署概要和最近活动。桌面端层级已改善，但移动端形成连续的等权白色矩形。

#### 整改

- 部署流程未完成时展示完整引导。
- 流程完成后收敛为一行状态摘要，不长期占据整张引导卡。
- 将四个指标收敛到一个 `MetricGroup` 表面，内部使用分隔线。
- 保留运行概览和访问入口两个主要内容区。
- 部署概要使用行列表，内部不再继续生成同权 Card。
- 最近活动使用 plain list 和分隔线。
- 移动端首先展示失败状态与下一步操作，其余信息允许折叠。

#### 涉及文件

- `web/src/pages/applications/application-overview-panel.tsx`

### 4.3 统一页面主操作层级

#### 现状

部分页面同一首屏同时出现两个实心主题色按钮，例如：

- 部署页：“新增部署配置”与“创建发布”。
- 通知页：“从模板创建”与“创建渠道”接近同等权重。

#### 整改

- 同一页面或 tab 默认只保留一个实心主操作。
- 部署配置已存在时，“创建发布”为主操作，“新增部署配置”为 outline。
- 无部署配置时，“新增部署配置”为主操作，发布操作 disabled 或隐藏。
- 通知页以“创建渠道”为主操作，“从模板创建”为 outline。
- 低频操作进入下拉菜单，但不得隐藏危险操作确认。

#### 涉及文件

- `web/src/pages/applications/application-deployments-panel.tsx`
- `web/src/pages/settings/NotificationsPage.tsx`
- 其他同时出现多个实心主题按钮的页面

### 4.4 账号页迁移到设置宽度体系

#### 现状

全局设置已经收窄到合理内容宽度，但账号页的昵称、头像 URL、语言等字段仍接近铺满 1500px 内容区，短内容输入框过长。

#### 整改

- 账号、安全设置、个人令牌、开发者应用和已授权应用统一使用 `SettingsShell`。
- 主要表单宽度控制在约 720 至 800px。
- 用户头像与身份摘要保留在顶部或独立概要区。
- 保存按钮与表单右边界对齐。
- DataList 类型的个人令牌和应用列表可以使用更宽内容区，不受表单宽度限制。

#### 涉及文件

- `web/src/pages/settings/AccountPage.tsx`
- `web/src/pages/settings/account-password-panel.tsx`
- `web/src/pages/settings/account-mfa-panel.tsx`
- `web/src/pages/settings/account-oauth-panels.tsx`
- `web/src/pages/access-tokens/AccessTokensPage.tsx`

### 4.5 收敛用户列表 Badge 数量

#### 现状

用户列表每行同时以胶囊展示角色、密码登录、两步验证和账户状态，造成状态视觉过载。

#### 整改

- 账户状态保留语义 Badge。
- 平台角色改为普通文字或更弱的中性标签。
- 密码登录和 MFA 合并为“认证”列，使用短文字、图标或两项紧凑状态。
- 高频正常状态不全部使用绿色；异常和需要处理的状态优先着色。
- 保留可访问名称和 Tooltip，不使用无说明的单独图标。

#### 涉及文件

- `web/src/pages/settings/UsersPage.tsx`

### 4.6 运营面板使用产品化失败状态

#### 现状

Grafana iframe 无法加载时直接显示浏览器原生灰色失败区域和文件错误图标，容易被理解为 Luna DevOps 页面损坏。

#### 整改

- iframe 加载前使用 `ToolViewportSkeleton`。
- 增加可配置超时检测或同源可观测的失败信号。
- 失败时显示平台 `ErrorState`，提供：
  - 重新加载。
  - 在新窗口打开。
  - 前往全局设置检查地址。
- 只有 iframe 确认加载后才展示工作区。
- 不能可靠判断跨域 iframe 失败时，至少提供覆盖层说明与“在新窗口打开”。

#### 涉及文件

- `web/src/pages/settings/OperationsDashboardPage.tsx`

### 4.7 调整移动端项目空间工具栏

#### 现状

390px 下搜索、范围、排序、排序方向和创建按钮分成多行，排序方向和创建按钮缺乏组合关系，“创建”文案离开上下文后语义偏弱。

#### 整改

建议结构：

```text
搜索项目空间
[与我相关] [最近使用]
[排序方向]              [创建项目空间]
```

或使用紧凑的筛选/排序弹出层，但必须：

- 排序字段与排序方向保持同一组合。
- 主按钮保留“创建项目空间”语义，空间不足时使用图标并提供明确 aria-label/Tooltip。
- 控件宽度和换行规则由 `PageToolbar` 提供，不在页面内逐项拼接。

#### 涉及文件

- `web/src/pages/projects/ProjectsPage.tsx`
- `web/src/components/common/page-toolbar.tsx` 或现有等价组件

### 4.8 优化移动端列表横向滚动

#### 现状

移动端项目空间列表出现明显原生横向滚动条，列表与分页之间被视觉切断。

#### 整改

- 为高频列表定义移动端保留列，避免只显示名称和操作时仍保留桌面最小宽度。
- 次要字段进入主单元格第二行或详情 Sheet。
- 必须横向滚动的复杂资源表格保留固有宽度，并提供轻量左右滑动提示。
- 细化滚动条样式，避免在默认静止状态形成粗横线。
- 固定操作列不得覆盖数据内容或 Debug 按钮。

#### 涉及文件

- `web/src/components/common/data-list.tsx`
- `web/src/pages/projects/ProjectsPage.tsx`
- 其他移动端高频 DataList 页面

## 5. P2：视觉精细化

### 5.1 移除应用市场重复标题

顶栏已显示“应用市场”，正文工具栏不再重复同名标题。可以：

- 改为“发现应用”并增加简短描述；或
- 完全移除正文标题，只保留搜索和筛选。

涉及文件：`web/src/pages/app-templates/AppTemplatesPage.tsx`。

### 5.2 调整顶栏密度

普通顶级页面的顶栏主要承载单个标题，当前高度略显空。评估将桌面高度从 64px 收敛至约 56px，同时：

- 保持移动端菜单触发器可点击面积。
- 项目/应用详情继续显示可点击名称链路。
- 不把大量业务按钮重新塞进顶栏。
- 页面 tabs 与标题之间仍保留明确层级。

涉及文件：`web/src/layouts/AppLayout.tsx`。

### 5.3 统一技术词汇与中英文混排

建立前端词汇表并收敛：

- Git Provider / Git 凭据。
- OIDC Provider / 准入策略。
- Group Claim、Scopes、Owner 等技术字段。
- 中英文之间的空格、大小写和复数规则。

原则：

- OIDC、Git、Scope 等技术专名可保留英文。
- 普通操作、状态和表头优先本地化。
- 同类页面保持一致命名模式，例如“创建 OIDC Provider”和“创建 Git Provider”。

涉及文件：

- `web/src/i18n/locales/zh-CN`
- `web/src/i18n/locales/en-US`
- 代码库与身份源页面

### 5.4 降低主题色视觉噪声

当前个人主题色会同时影响按钮、tabs、分页、图标和部分强调文字。在高饱和主题下容易形成过多彩色焦点。

整改：

- 实心主按钮继续使用 `primary`。
- tabs 使用文字色和细指示器，不使用大面积主色背景。
- 分页选中使用 subtle 背景与清晰边框。
- 图标强调优先使用 `primary-text`。
- 同一首屏只有一个主色实心按钮。
- Luna Logo 和固定品牌图形继续保持品牌蓝，不随个人主题漂移。

涉及文件：

- `web/src/styles/brand-themes.css`
- `web/src/components/ui/tabs.tsx`
- `web/src/components/ui/pagination.tsx`
- `web/src/components/ui/button-variants.ts`

### 5.5 改善数据较少页面的重心

代码库、镜像站、集群和通知等页面在数据很少时，上方只有一个横向大容器，下方大面积留空。

整改原则：

- 空列表容器保持紧凑，不为占满页面设置固定高度。
- 有真实说明价值时可增加使用指引或文档入口。
- 不为了填空增加无业务意义的统计卡片。
- 配置类资源页可以评估 `max-width: 1200px` 至 `1400px`，普通数据列表仍允许全宽。

## 7. 建议继续加入开发规范的规则

现有 `AGENTS.md` 已包含页面类型、`PageShell`、语义布局、状态色、间距和 `DataList` 外壳要求。后续建议仅补充以下尚未覆盖的规则：

### 7.1 加载与空状态

- 列表、概览、设置和工具工作区必须使用对应结构化 skeleton；禁止使用大容器只展示一行“加载中”。
- `total === 0` 时不得展示无意义页码和翻页器；可行动空状态必须提供明确下一步。

### 7.2 主操作

- 同一页面或 tab 默认最多一个实心主色主操作；其他同级操作使用 outline、ghost 或菜单。

### 7.3 概览异常优先级

- 看板和概览中的失败、不可用和待处理状态必须在摘要层使用语义 tone，并提供处理入口；不得只在下级列表中显示异常 Badge。

### 7.4 移动端渐进披露

- 桌面端超过 4 个筛选字段时，移动端必须使用筛选 Sheet、弹出层或等价渐进披露方式，不得简单纵向堆叠全部筛选。
- 高频移动端列表应定义保留列，不能默认依赖桌面表格横向滚动完成适配。

### 7.5 浮层安全区

- fixed/sticky 悬浮控件不得覆盖主操作、表格固定列、分页、toast、Dialog 或 Sheet；必须在桌面和移动端验收安全区。

上述规则应在对应公共能力落地后再加入 `AGENTS.md`，避免规范先于可复用实现。
